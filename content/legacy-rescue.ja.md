## SMBC FOX FX取引システムモダナイゼーション

### 背景

20年以上の蓄積複雑性を持つプラットフォームのレガシー発掘調査、移行工学、運用救出を必要とする高信頼性銀行コアシステムプロジェクト。クライアント: SMBC。システム: FOX外国為替/金融デリバティブ取引プラットフォーム。環境: DXC Technology。

### 真の課題

これはインフラ移行ではなかった。システム発見の問題であった。

プラットフォームは20年以上継続的に進化してきた。ドキュメントは欠落、陳腐化、または実システムと矛盾していた。ビジネスロジックはシェルスクリプトやコンパイル済みバイナリとしてのみ存在していた。システムが理解されるまで、移行は安全に進めることができなかった。

制限された銀行環境内でgrep、vi、シェルユーティリティのみを使用 — IDE無し、ソース索引無し、自動分析ツール無し。

## VC6リンカ手術: 時間軸超越ABI互換性

### 背景

最新の暗号要件が古いABI制約と衝突するレガシー銀行基盤（Windows XP / Visual C++ 6.0）。大規模な再配布パッケージやSxSマニフェストクラッシュ無しに最新暗号プリミティブを注入することが課題。

### 技術的課題

- VC6とVS2015の静的/動的ライブラリ混合時のLNK2005シンボル衝突
- Windows XPのSxSマニフェストチェックによる断続的アプリケーションクラッシュ
- OSレベルのセキュリティフィルタがUPX圧縮ファイルをフラグ
- Version 391の「不可視デッドロック」— ログ無し、クラッシュ無し、フリーズしたUIのみ

### エンジニアリング判断

- **シンボルハイジャック:** 最終リンク時に標準 `MSVCRT.LIB` と `LIBCMT.LIB` を無視、必要なシンボルを手動再リンク
- **/MTフラグですべて静的リンク**、その後不要シンボルを削除
- **マイクロペイロードアーキテクチャ:** 33KB DLLに最新セキュリティロジックをカプセル化
- **UPX削除:** ネットワークフィルタ通過のために圧縮を除去、100%スループット達成
- **バージョン履歴の二分探索**でVersion 391のグローバル `send_mutex` に障害を特定
- 同期をスレッドアフィニティキューにリファクタリング

### 成果

| 指標 | 結果 |
|---|---|
| 配布フットプリント | 1.3MB → 230KB（82%削減） |
| マイクロペイロードDLL | 33KB |
| 橋渡ししたコンパイラギャップ | 20年（VC6 1998 → VS2015） |
| ネットワークフィルタスループット | 100%（UPX削除後） |
| デッドロック解決 | スレッドアフィニティキュー |
| ビジネスインパクト | 国営銀行端末ネットワーク安定化 |

## 破られざる橋の工学: 言語間インターフェースの強化

### 背景

エンタープライズシステムで最も脆弱な層、すなわちC++とマネージド言語（C#/.NET、Java/JNI）の境界を強化するシニアアーキテクトの現場マニュアル。銀行ATM端末、組込みHSM、国家標準暗号SDK（SM9/IBC）、Windows XP上のInternet Explorer内ActiveXコントロールにわたる10年以上の言語間橋渡し構築経験。

### 技術的課題

- 異なるCRTアロケータ（VS2010 DLL + VC6 MFCアプリ）によるヒープ破損
- 呼び出し規約不一致（`__cdecl` vs `__stdcall`）によるサイレントスタック破損（`0xC0000005`）
- 非決定論的GCファイナライゼーションによるUSB HIDデバイスのハードウェアハンドルデッドロック
- DLL境界をエスケープする未キャッチC++例外がホストプロセス（IE、Tomcat）を強制終了
- クロスコンパイラタイムワープ: 最新C++（VS2010、Boost 1.56、Poco 1.4.6）をWindows XP / IE6 / VC6ホストに注入
- JNIで30年タイムスタンプロールバックを引き起こす `time_t` ビット幅不一致（1375891200 → 424567760）

### 5つのエンジニアリングメカニズム

**メカニズム1 — 次元削減:** DLL境界を越えて複雑なC++オブジェクトを渡さない。`const char*`、Protobufシリアライゼーション、整数エラーコードに削減。

**メカニズム2 — 呼び出し規約強制:** 明示的 `__stdcall` 宣言 + .DEFファイルシンボルピン留め。一貫性は慣習に勝る。

**メカニズム3 — 明示的ライフサイクル:** Open → Authenticate → [Logic] → Close を強制。ファイナライザからの自動リソース解放を全て除去。~130-140操作でのポートデッドロックを解消。

**メカニズム4 — 例外サイロ:** 全スレッド関数境界でのトータルキャッチハンドラ。ベースDLLとの衝突を避けるためエラーコードは16から開始。

**メカニズム5 — SWIGタイプマップ:** タイムスタンプ安全性のための明示的 `time_t` → `jlong` マッピング。暗号ペイロードのためのバイナリセーフ `(char*, size_t)`。

### 成果

| 指標 | 結果 |
|---|---|
| DLLサイズ | 931 KB単一DLL、ゼロ外部依存 |
| ポートデッドロック | ~130-140 ops → 1,513ループ、4,500+認証サイクルでゼロクラッシュ |
| Device Open | 10ms |
| Authentication | 70ms |
| Pocoロギングオーバーヘッド | 20ms/エントリ → printf/OutputDebugStringで0.4ms |
| ログ圧縮 | 45MB → 1.5MB .gz |
| 同時スレッド | 100 `boost::thread` で障害無し達成 |

## レガシー産業用SCADAプラットフォーム安定化（Qt 4.7 / VS2003）

### 背景

2017年頃、電力・風力エネルギー業界で使用される大規模産業用SCADAプラットフォーム。Qt 4.7.3とVisual Studio 2003上に構築され、厳しい運用制約のあるレガシーWindows環境で稼働。リアルタイムテレメトリ、バックエンドサービス、産業用通信、長期稼働クライアントを含む大規模プロダクションシステムの一部。

### 技術的課題

- ビルドパスが破損したQt 4.7.3 / VS2003ツールチェーン依存
- 緩く管理されたモジュールと部分的なビルドドキュメント
- ハードコードされたパスと脆弱なデプロイ前提
- レガシーOCXとFormula Oneコンポーネント依存
- Qt4 / Qt5互換性ギャップ
- 長期運用下でのWindows GDIハンドル枯渇
- 広範なリファクタリングを許容しない本番システム

### エンジニアリング判断

- 破損したQt 4.7.3ビルドパスを復旧、MOC / QMake互換性を修復
- VS2003リンカおよびツールチェーン問題をデバッグ
- Qt再描画ライフサイクルと相関するGDIハンドル枯渇を調査
- Python / PyQt / 埋め込みブラウザを使用した代替可視化パスを設計

### 成果

- プロット容量を~30,000ポイントから~100,000ポイントに増加（3倍改善）
- GDI主体レンダリングパスでのクラッシュリスク低減
- 反復可能な本番修正のためのビルドチェーン復旧
- 完全書き換えを強制しない部分モダナイゼーション

## 動的パスワード認証によるオフラインATMセキュリティロック

### 背景

ATMメンテナンスおよび制御アクセスワークフローで使用される電子ロックのセキュリティ・通信レイヤを設計。継続的なネットワークアクセスがない環境でも動作する必要があった。デバイスはリソース制限のある組込みハードウェア上で稼働。

### 技術的課題

- パスワードフローはオフラインでも安全である必要があった — 時間ベースOTPは有効期間内にリプレイ可能
- 制約のある組込みハードウェア上の暗号パフォーマンス
- C++ DLL、HID通信、Python、Java、C#ラッパー間のホスト-デバイス信頼性
- マルチスレッドリクエスト処理。トランスポートバグとスレッドタイミングがセキュリティ問題に

### アーキテクチャ

- **暗号構成要素:** ハッシュベース認証、PSKフロー、AES対称暗号化、ECDH鍵交換
- **ホスト側:** DLL境界を通じて公開されるC++アルゴリズムレイヤ、Python/Java/C#用SWIGラッパー
- **デバイス側:** STM32クラスARMハードウェア、HID通信プロトコル
- **状態保持認証:** 各パスワードは直前の閉状態に依存し、リプレイを防止

### 成果

| 指標 | 改善前 | 改善後 |
|---|---|---|
| ロック通信時間 | ~10-16秒 | 2-3秒 |
| 遅延レポートフロー | ~6.1秒 | 0.9秒（6.8倍） |
| HIDパケットレイテンシ | ~1.6ms/パケット | ~1.0-1.1ms/パケット |
| リクエストパスHIDパケット | 11パケット | 3パケット |
| ARM上のEC初期化 | — | ~0.75秒、~10KBメモリ |

## レガシーセキュリティ強化: 3DES/AES救済レイヤ

### 背景

2014年、大手国営インフラプロバイダが深刻なセキュリティ危機に直面。全国に展開された数千のレガシー端末ユニットが、最新コンプライアンス基準を満たさない老朽化3DES暗号スキームを使用。ハードウェアは「凍結」状態 — ファームウェア更新は高リスク、完全交換費用は数百万ドルと見積もられた。

### 技術的課題

- 3DESは最新のブルートフォースおよび中間一致攻撃に対して脆弱
- パケット構造への変更は数千ノードのレガシーパースロジックを破壊
- 完全書き換えは「移行麻痺」につながる
- レガシープロトコルの固定サイズペイロード制約
- フェーズドアップグレードのためのゼロダウンタイム要件

### アーキテクチャ

- **二重パス暗号化:** ハートビート/低感度テレメトリにはレガシー3DESパスを維持。高感度制御コマンドには強化AES-256 GCMパス
- **暗号プロキシ/サイドカー:** レガシー端末とバックエンド間のトラフィックをインターセプトし再暗号化
- **鍵ローテーション管理:** 一意の物理端末ID + 集中HSM
- **フェーズド移行:** 救済レイヤが移行期間中に3DESとAESの両方をシームレスに処理

### 成果

| 指標 | 結果 |
|---|---|
| 移行 | ゼロダウンタイム、フェーズドシームレス移行 |
| コスト削減 | ~$15.4Mのハードウェアリコールおよび人件費回避 |
| エビデンス耐久性 | 認証暗号により99.9999%（シックスナイン） |
| アーキテクチャ | 非侵襲的な暗号プロキシ/サイドカー |

## 12KBシステム主権: MIRACLメモリ戦争

### 背景

2013年、超高セキュリティ商用暗号デバイス（UKey）の基盤セキュリティレイヤ構築。コアハードウェア: 国産SSX45チップ、C\*Core C340プロセッサ（32ビットRISC）、12KB SRAM。この制約内で複雑なECCおよび中国国家SM9アルゴリズム（Tate Pairingベース）を実行する必要があった。

### 技術的課題

- 合計12KB SRAM — MIRACLライブラリ初期化で8K-9.5K消費、残り3KB未満
- MMU無し、最新OS保護無し — ヒープ断片化は時限カーネルパニック
- MicroLib `time()` が8時間ずれた値を返す（28800秒タイムゾーンドリフト）、PCジャンプを引き起こす
- コンパイラ `-O0` が誤ったSHA-256スタックフレームを生成
- ハードウェアゴースト `0xE3`: データバイトがアダプタのハードコード「Global Reset」コマンドと一致（インバンドシグナリング）
- 呼び出し規約不一致がz=7で `0xC0000005` を引き起こす（28バイト累積スタックリーク）
- FreeLibraryトラップ: `boost::thread` ログスレッドがDLLアンロード時に非同期に強制終了

### エンジニアリング判断

- **動的割り当ての全面禁止:** `#define MR_STATIC 32` でMIRACLを全スタックモードに強制
- **手動mrkill挿入:** 全Big Number操作後に手動配置メモリ解放
- **MicroLib timeの放棄:** 純粋数学的グリニッジ標準時を記述。タイムゾーン用に独立メモリを確保
- **USB HIDカモフラージュ:** ATMドライバホワイトリストをバイパスするため正当なキーボードVID/PIDを借用
- **ドライバレイヤでの強制エスケープと物理ステッピング**で `0xE3` ハードウェアゴーストを抑制
- **統一 `__stdcall` エクスポート** + `[UnmanagedFunctionPointer(CallingConvention.Cdecl)]` で呼び出し規約修正

### 成果

| 指標 | 結果 |
|---|---|
| メモリフットプリント | 10KB → 576バイト（94%削減） |
| SRAM使用率 | 99.9%、100%決定性 |
| 運用記録 | 10年以上ゼロダウンタイム、ゼロインシデント |
| 1-Wire読み取りタイムアウト | 物理レッドライン40ms（30msは失敗、50msは動作） |
| SHA-256 | 2倍のエネルギーコストにもかかわらずSHA-1より採用 |

## WinCEプラットフォームエンジニアリング: OpenSSL + SM9暗号スタック

### 背景

2012-2013年頃、北京久其政府（暗号ロック部門）チームの一員として銀行ATMセキュリティロックおよび金融端末組込みデバイスを構築。ARMベースWindows CEプラットフォームに完全な暗号スタック（OpenSSL、MIRACL、中国国家SM9 IDベース暗号標準）をもたらすことが課題。

### 技術的課題

- 標準ツールチェーン互換性のないARMV4Iアーキテクチャ上のWinCE 5.0/6.0
- OpenSSL 0.9.8w/1.0.0/1.0.1cはx86からARMへのクロスコンパイルが必要
- 初期OpenSSLリンク時に116の未解決シンボル — CRT不一致、ソケットAPI差異（WS2.dll）、ARMランタイム関数欠落
- PEヘッダーバージョン非互換性によるDLLロード失敗
- MIRACLライブラリにx86アセンブリ最適化があり、純粋Cに置き換えが必要（`MR_NOASM`）
- ARMランタイム関数ギャップ: `__rt_udiv64by64`、`__rt_sdiv`、`__rt_urem64by64` がターゲット環境に不在
- 乱数生成器クラッシュ、ゼロ除算例外、複数ライブラリバージョン間のスレッド安全性分析

### エンジニアリング作業

- **クロスコンパイル環境:** ARMV4I上のWinCE 5.0/6.0をターゲットとするEVC4 / VS2005 / VS2008ツールチェーンをセットアップ
- **OpenSSL移植:** CRT互換性、ソケットレイヤ適応、ARMランタイム関数橋渡し、PEヘッダ互換性にわたる116+のリンカ/ランタイム問題を解決
- **MIRACL移植:** アセンブリパスを無効化（`MR_NOASM`）、x86最適化プリミティブをECC、Big Number、Pairing演算の純粋C実装に置き換え
- **SM9暗号スタック:** IBCアルゴリズム工学、WinCE適応、ARM最適化を含む完全SM9 IDベース暗号ライブラリを移植
- **システムレベルデバッグ:** dumpbin/dependsを使用したDLL依存関係分析、PEヘッダフォレンジック、静的/動的ライブラリ競合解決（LNK2005/2019/2001）

### 成果

- OpenSSL 0.9.8w/1.0.0/1.0.1cをARM WinCEプラットフォームに移植成功
- ARM上でアセンブリ依存無しにECC + Pairingを実行するMIRACL
- WinCE組込み端末で完全SM9暗号スタックが動作
- 本番銀行セキュリティ端末および組込み暗号デバイスに適用
- 希少スキル組み合わせ: VC++ + WinCE + ARM + 暗号技術 — 市場で極めて入手困難になった組み合わせ

*本付録は、レガシー救出、VC++相互運用、プラットフォームモダナイゼーション、組込みシステムロールに最も強いシグナルを提供するエビデンスを優先する: プラットフォーム発見、ビルドチェーン復旧、リンカ/ABI工学、言語間橋渡し、組込みセキュリティ工学。*
