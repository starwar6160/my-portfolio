## 挑戦：ベアメタル上におけるIBCの「恐怖」

2013年当時、**ARM Cortex-M3 (STM32F2)** 上で **SM9 (Identity-Based Cryptography)** 規格を実装することは、多くのエンジニアにとって「不可能」と考えられていました。標準的なECCとは異なり、SM9は**ペアリング（Pairing-friendly curves）**を必要とし、256ビットの座標を1536ビットの構造に拡張する大規模な体拡大（Field Expansion）を伴うためです。

標準的な暗号ライブラリ（OpenSSL, MIRACL）はサーバー向けに設計されており、RAMの1バイトが貴重な組込みシステムには適していません。私の任務は、肥大化した実装を徹底的に削ぎ落とし、セキュアUSBKeyや**オフラインATMダイナミックロック**システムに求められる **8KB RAM以下** の制約内に収めることでした。

---

## 1. 外科的アーキテクチャ：「メタデータの肥大化」の除去

2013年の監査で最初に発見したのは、メモリフットプリントの70%がアルゴリズムそのものではなく、**インフラストラクチャ**によるものだということでした。
*   **ASN.1 形式のデコード**: ヒープ空間を大量に消費する要因。
*   **OpenSSL への依存**: マイクロコントローラには不要な数千行のコード。

### 戦略：「純粋アルゴリズム」の抽出
私は実装を**ベアメタルCプリミティブ**へと絞り込む作業を主導しました。複雑なASN.1を廃止し、**Protobuf** または静的な固定サイズ構造へと切り替えました。これにより「メモリドリフト」のリスクを排除し、ヒープのフットプリントを60%以上削減しました。

---

## 2. メモリの制御：`mralloc.c` によるブレークスルー

8KBの制限を達成するために、標準の `malloc` に頼ることはできませんでした。私は基盤となる MIRACL メモリアロケータ（`mralloc.c`）を改造し、ピーク時のヒープ使用量を厳密に統計・追跡する機能を組み込みました。

```c
/* 
 * 2014年 8KB RAM最適化作業からのスニペット
 * 目的: ECDH/SM9 鍵交換におけるピークメモリの制限
 */
static size_t peak_memory_usage = 0;
static size_t current_usage = 0;

void* mr_alloc(size_t size) {
    current_usage += size;
    if (current_usage > peak_memory_usage) {
        peak_memory_usage = current_usage;
    }
    // リソース制約のあるターゲットに対する厳格な執行
    if (peak_memory_usage > MAX_RAM_BUDGET_BYTES) {
        log_critical("メモリ制限違反！ 予算を超過しました: %zu", peak_memory_usage);
        return NULL; 
    }
    return standard_malloc(size);
}
```

**結果**: 徹底的な洗練の結果、フルスタックのECDHでピーク時のアロケーションを **9,300バイト** まで抑えることに成功しました。「科学者」としての視点を持ってメモリを外科的に管理することで、汎用ライブラリを遥かに凌駕する性能を引き出せることを証明しました。

## 3. ハードウェアに固定された決定論

暗号アルゴリズムの強さは、その**エントロピー（複雑性）**に依存します。標準的なハードウェア上のソフトウェアベースの乱数生成器（PRNG）は予測可能であるため、脆弱性が残ります。

私は以下の**ハードウェア・アンカー（根拠）に基づいたセキュリティ**への移行を設計しました：
1.  **TRNG の統合**: 安全チップ（FMCOS）やハードウェア暗号化カードの **真の乱数生成器（TRNG）** から直接シークエンスを取得するように、コアの乱数生成機能を書き換えました。
2.  **カーネルレベル・ドライバ**: 「シリコンの真実」をユーザー空間のSM9ライブラリに橋渡しするためのJNIおよびLinuxドライバを実装。

## 4. 「脏手（Dirty Hands）」の証明

ホワイトボード上でアーキテクチャを議論するだけではなく、私は2014年の大半をラボで過ごし、**座標変換（Coordinate Transformation）**の問題を解決しました。SM9において6倍の体拡大は、曲線上の1点が多大なメモリブロックを占めることを意味します。

私は **ヤコビ座標（Jacobian Coordinates）** の数学的処理を自ら最適化して一時オブジェクトを最小限に抑え、中間計算がSTM32などのタイトなRAM内に確実に収まるようにしました。

```text
[ 高レベル暗号ライブラリ ]  ⮕  [ メモリの肥大化 (ASN.1, ヒープ断片化) ]
           │
           ▼  外科的除去 (2014年監査)
           │
[ 手作りCアルゴリズム ] ⮕  [ 9,300バイトの静的ヒープ (mralloc.c) ]
           │
           ▼  ハードウェアへの直接フック
           │
[ STM32F2 (Cortex-M3) ] ⮕  [ ハードウェアTRNG ] ⮕ [ 銀行級の堅牢性 ]
```
