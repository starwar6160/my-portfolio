# 12KBの戦壕にて：システム主権とセキュリティ防衛の30年回顧録

### イントロダクション (The Hook)

Web3、DePIN（分散型物理インフラ）、あるいは高セキュリティが求められる組み込みシステムの開発現場において、リソース制約とセキュリティの整合性の境界線は常に極めて重要な課題です。モダンな抽象化レイヤーが高レベルの保護を提供する一方で、システムの真の主権は依然としてレジスタやメモリサイクルのレベルで決定されます。

メモリ安全性やリエントランシー防御の本質的な解を求める際、その答えは巨大なアーキテクチャ図ではなく、極めてシンプルな低レイヤの制御に収束することがあります。

`#define MR_STATIC 32` と `mrkill`。

クラウドネイティブという、リソースが潤沢な環境で積み上げられた抽象化レイヤは、時にシステムの物理的な制約を隠蔽してしまいます。12KBのRAMという極限状態を経験した技術者は、物理的な境界線に対する鋭い嗅覚を備えています。真のセキュリティとは、APIゲートウェイを積み上げることではなく、**全CPU命令、全メモリサイクル、そして物理回路の通信パルス幅に至るまでの絶対的な支配権を握ることにあるのです。**

今日、私は2012年から2017年にかけての古びた業務ログを紐解き、コンパイラが嘘をつき、標準Cライブラリが裏切り、ハードウェアが「ゾンビ」のように蘇る暗黒の中世へと皆さんを連れ戻そう。汗と怒りにまみれたコードの断片の中に、アーキテクトがいかにしてシステムレベルの混沌から血路を切り開き、侵すべからざる「システム主権」を確立したかが見えるはずだ。

**現代のWeb3ハードウェアウォレット、DePIN（分散型物理インフラネットワーク）、あるいはミッションクリティカルなIoTデバイスを構築する企業にとって、これは単なる技術的な回想録ではない。ハードウェアのリコール費用を数千万単位で節約し、壊滅的な資産流出を防ぐための「商用ブループリント」である。**

## 第1章：資源隔離の「禁欲主義」 (Zero-Sum Memory)

スマートコントラクトのバイトコードが数十KB増えるだけでコストを嘆く現代において、真の物理的レッドラインとは何かを教えよう。

2013年、私たちは超高セキュリティの商用暗号デバイス（UKey）の基盤セキュリティ層を構築していた。採用したハードウェアは国産のSSX45チップ、心臓部はC*Core C340プロセッサコア（32ビットRISCアーキテクチャ）。このチップが私に突きつけた「贈り物」は、わずか12KBのSRAMだった。

現代のプログラマが `var buf = new byte[1024]` と打ち込むとき、鼓動が速まることなどないだろう。しかしSSX45において、1KBは領土の8%に相当する。この極限の制約下で、私たちは複雑な楕円曲線暗号（ECC）と、Tate Pairingに基づく国密SM9アルゴリズムを動かさなければならなかった。

標準的な動的ヒープ割り当て（Heap Allocation）を使用すれば、MIRACL暗号ライブラリの初期化だけで一度に8Kから9.5Kのメモリが飲み込まれる。残された3KB足らずでは、TCPハンドシェイクの半分も支えきれない。さらに致命的なのは、MMU（メモリ管理ユニット）も現代的なOSの保護もないベアメタル環境において、`malloc` と `free` が生み出すメモリ断片化（フラグメンテーション）は、システムを死に至らしめるカーネルパニックの時限爆弾だということだ。

私は、冷徹で独裁的な決断を下した。**「動的割り当ての全面禁止、スタック（Stack）禁欲主義の徹底」**である。

コンパイル時に以下のマクロを強制的に注入した：

```c
// 1024ビット鍵を扱う32ビットプロセッサにおいて、MIRACLライブラリのすべてのmallocヒープ割り当てを廃止し、
// すべての状態をスタックフレームに押し込める
#define MR_STATIC 32 
```

この一行により、すべての巨大数（Big Number）のメモリ割り当てをスタックへ再指向し、管理下に置いた。しかし、それだけでは不十分だった。Tate Pairingの `ecap` 演算は、膨大な数の中間変数を生成する。関数の終了を待ってスタックを解放するのでは、12KBという薄氷の上のスタックポインタは即座に突き破られてしまう。

私はメスを握る清掃員のように、C++の優雅なRAIIや演算子オーバーロードを完全に捨て去り、古風な純粋C言語の関数呼び出しへと退行した。そして、巨大数の乗算や加算が行われるたびに、手動で `mrkill` 命令を注入した。変数がその価値を失った瞬間、そのCPUサイクル内で「原子レベルの破壊」を遂行するためだ。

> *「ecap関数内部で2つのmrkillを使用し、不要になった一時変数（巨大数）を即座にクリアした。結果、メモリ消費を25221から24645へと削減（576バイト、あるいは144バイト×4ブロックの節約）に成功した。」* —— 2014年の原始ログより

最終的に、凄惨な肉弾戦を経て、私は大規模な署名操作のメモリフットプリントを10KBから驚異の576バイトへと圧縮した。Solidityコンパイラのスタックの浅さを嘆く現代のWeb3ギークたちは、真の「ゼロサム・メモリゲーム」が何たるかを知らない。メモリ使用率を99.9%まで追い込み、なおかつシステムが100%の決定論で動作し続けるとき、その絶対的な支配感こそが、アーキテクトがシステムに刻み込む至高の「主権」なのだ。

**商用的な視点で見れば、暗号演算をこれほど低コストなシリコンに詰め込む能力は、ユニットあたりの部品コスト（BOM）を劇的に削減する。制度級の暗号強度を維持したまま、安価なノードを大量展開することを可能にし、ハードウェアメーカーに強固な競争優位性をもたらすのである。**

## 第2章：不可視の裏切りとの闘い (Compiler & C Runtime)

クラウドネイティブの時代、開発者はコンパイラ、OS、低層ライブラリを神のように崇め、標準APIを呼び出しさえすれば物理ロジックは完璧に動作すると信じ込んでいる。しかし、12KBの戦壕において、神は嘘をつく。

### 時空共識の危機と MicroLib 28800秒の背信

暗号コンポーネントには、リプレイアタックを防ぐための厳格なタイムスタンプが必要だ。しかし、ベアメタルARMでのテスト中、スマートロックとサーバ間の時間の合意が、常に正確に「8時間」ズレるという深刻な断絶に直面した。Cコードで直接UTCオフセットを計算しようとした試みは、凄惨な「法医学レベル」の脱線事故を招いた。

ログが出力した鮮明な「時間衝突データ（Time Collision Data）」を見てほしい：

```text
// PC側（標準環境）の正常な時間取得
INPUT time() = 1390468440  (日本時間 2014-01-23 18:14:00に相当)

// ARMベアメタル側が算出した出力
OUTPUT time() = 1390497240
```

推論の結果、`1390497240 - 1390468440 = 28800秒`。これは正確に8時間、すなわち中国標準時（UTC+8）のオフセット量だった。

8時間足りないことを突き止めた私は、Cコードに `time_t += 28800;` という単純な補正ロジックを書き加えた。しかし、この何の変哲もない加算代入文をターゲットチップにデプロイした瞬間、システムは**「ハイパースペースへと跳んでしまった（暴走した）」**のである！

> *「たった今、ARM環境向けにtime関数の出力に8時間を加算するよう修正したコードが、梁技術者のボード上で暴走した。何の変哲もない代入文から、私が書いた全く無関係な別の関数へと実行がジャンプしてしまった。現象としてはバッファオーバーフロー攻撃に近い……」*

**失敗経路の硬核な推論：**
なぜ、単に `time_t` に `28800` を足しただけで、システムがクラッシュし、命令境界が破られたのか？

並のプログラマなら型キャストのミスを疑うだろうが、私は `DWT_CYCCNT` 命令を使用してCPUサイクルレベルの追跡を行った。真実は極めて暗いものだった。ARMコンパイラ（Keil MDK）に付属する極小ランタイムライブラリ `MicroLib` は、フットプリントを1.2KBまで削ぎ落とすために、`localtime` における時区解析ロジックを完全に去勢していただけでなく、標準Cのスタックアライメント保護さえも省略していたのだ。

MicroLibの公式マニュアルの隅に、私は以下の「冷酷な判決文」を見つけた。
> *"Microlib is not compliant with the ISO C library standard... Locales are not configurable. The default C locale is the only one available."*

この去勢されたライブラリにおいて、`tm` 構造体と低層の時間計算ロジックは静的な疑似割り当てに書き換えられていた。私たちが強引に `+28800` を注入し、32ビットのCortex-M3上で64ビットの整数格上げ（Integer Promotion）を行った際、メモリのアライメント要求が粉砕されたのだ。コンパイラが生成した `STRD`（ダブルワードストア）命令は、8の倍数ではない未整列のスタックアドレスに64ビットのタイムスタンプを書き込もうとした。

結果は壊滅的だった。この書き込みは時間構造体を上書きしただけでなく、その境界を越え、現在のスタックフレームに保存されていた**リンクレジスタ（LR）**をも破壊した。関数がアセンブラレベルの `BX LR`（戻り命令）に達したとき、プログラムカウンタ（PC）には `28800` の上位バイトで汚染されたデタラメなアドレスがロードされた。CPUは瞬時にメモリ上の全く無関係な領域へと跳躍した。これはまさに悪意あるバッファオーバーフロー攻撃そのものだが、犯人はARM公式の標準ライブラリだったのである！

いわゆるANSI Cのクロスプラットフォーム互換性など、過酷なベアメタルハードウェアの前では紙屑に等しい。時間の制御権を取り戻すために、私たちはMicroLibの時間関数を完全に放棄し、グリニッジ標準時に基づく純粋な数学的推論アルゴリズムを自作し、物理的なセキュリティゾーンに独立したメモリブロックを確保して時区補正を格納しなければならなかった。OSというセーフティネットのない荒野では、メモリ管理だけでなく、時間の目盛りさえも自ら定義する必要があるのだ。

## 第3章：時序と信道の物理的絞殺 (HID Tunnel & In-Band Signaling)

コードレベルの裏切り以上に凄惨な戦いは、物理ハードウェアの境界線上で繰り広げられた。銀行のATMは高度にカスタマイズされたWindows XPで動作しており、そこには極めて厳格な「ドライバ・ホワイトリスト」が存在した。

**なぜ、従来のUSB-シリアル変換を使用しなかったのか？**
当時のテストでは、標準的なMSVC 10.0ランタイムや一般的なUSBシリアルドライバ（`.sys`）をATMシステムに注入しようとするたびに、カーネルレベルの能動的防衛機能が作動した。これはビジネスプロセスの連鎖的な停止を招くだけでなく、頻繁にブルースクリーン（BSOD）を引き起こした。銀行側は即座に拒否権を発動した。「いかなるサードパーティ製ドライバのインストールも厳禁とする」。

生き残るために、私たちは極端な「次元削減攻撃」を仕掛けた。**「擬装」**である。仮想シリアルポートを完全に捨て去り、システムに標準搭載されている「ヒューマン・インターフェース・デバイス（USB HID）」プロトコルを介してデータをトンネリングさせたのだ。どんなシステムであっても、マウスやキーボードを接続することは天然の「免駆（ドライバ不要）」だからだ。

```c
// システムのハードウェアホワイトリストを回避するため、
// 合法的なキーボード/マウスメーカーのVendor IDを借用し、HIDプロトコルトンネルを実行する
#define HID_VID 0x0483
#define HID_PID 0x5710
```

しかし、高並列の暗号通信をマウスやキーボードのポーリング信号に化けさせたことで、深刻な物理時序（タイミング）の裂傷が生じた。

### ミリ秒とマイクロ秒の乱闘：30msの生死線

底層の暗号ロックは1-Wireプロトコルを使用していた。1-Wireは時序に対して極めて過酷だ。暗号ロックを検証する際、戻ってくるデータは以下のようになる： `0A 4D 33 19 00 00 00 D5`。

これは適当なデータではない。HEXメッセージの最初の7バイトは、複雑なCRC8チェックを厳格に通過しなければならない。Maximのアプリケーションノート27を紐解き、私たちはデータの歪みに対抗するために極めて精緻な多項式チェックを実装した。

```c
// 1-Wire iButtonの底層時序保護のため、多項式 X^8 + X^5 + X^4 + 1 を実装
// メッセージ 0A 4D 33 19 00 00 00 に対して、算出されたCRC8が厳密に 0xD5 と一致しなければならない
uint8_t crc8( uint8_t *addr, uint8_t len) {
    uint8_t crc = 0;
    for(uint8_t i = 0; i < len; i++) {
        uint8_t inbyte = addr[i];
        for (uint8_t j=0; j<8; j++) {
            uint8_t mix = (crc ^ inbyte) & 0x01;
            crc >>= 1;
            if (mix) crc ^= 0x8C; // 多項式の逆順表現
            inbyte >>= 1;
        }
    }
    return crc;
}
```

1-Wireプロトコルで1ビットを書き込むには、電圧をLowに引き下げ、正確に60マイクロ秒（μs）保持する必要がある。しかし、外側のUSB HIDプロトコルにおける物理エンドポイントのポーリング限界は「1ミリ秒（ms）に1回」だ。この巨大な周波数の溝により、初期の通信は完全にデッドロックに陥るか、`0xFF` のゴミデータを返すだけだった。

仮想化マッピングでカプセル化されたインターフェースにおいて、文字送信間隔とリードタイムアウトの設定は、文字通り「生死を分ける」決定事項となった。

**物理的レッドラインのデータ：**
1-Wireのマイクロ秒信号がUSBのミリ秒ポーリングの中で失われないよう、私たちはコードに「ビジーウェイト」という最も不格好だが最も効果的なパッチを施した。
> *「後に判明したことだが、最初に2つのC1を送信し、文字間に少なくとも80msの間隔（50ms以下では失敗）を空けなければ、iButtonアダプタは正常に反応しない。」*

リードタイムアウト側では、`cProfile` を使用して `1121回` に及ぶシリアル入出力のオーバーヘッドを特定した。私は低層の物理的許容限界を攻撃的にテストし始めた。タイムアウトを200msから50ms（成功）まで圧縮し、次いで30ms（失敗・中断）を試し、最終的に **40ms** という絶対的なレッドラインに釘を刺した。

**なぜ30ミリ秒は謎の失敗を遂げたのか？**
これはシリコンチップのコンデンサ充放電という深層の物理推論を必要とする。アダプタはUSBパケットを解凍して1-Wireの電圧レベルに変換する。64バイトのブロックを送信するには、物理層で電圧を512回上下させる必要がある。この512回の電圧ジャンプに加え、マイコンの割り込みオーバーヘッドを含めると、物理的なシリコン上での総所要時間は**絶対に35ミリ秒を下回ることはない**。私たちが上位のWindows APIでタイムアウトを30msに強制したとき、物理コンデンサの充電すら終わっていなかったのだ！OSカーネルは正確に `IRP_MJ_READ` のキャンセル命令を発行した。この瞬間、ハードウェアアダプタのTX FIFOはまだ1-Wireバスにデータを吐き出していたが、ホストPCは一方的に通信契約を破棄していた。この「物理的な電気よりも速く走ろうとするコード」のズレが、ハードウェアの状態機を一瞬にして粉砕し、以降のパケットをすべてドロップさせたのである。

ベアメタルのトンネリングにおいて、優雅な非同期 `await` など存在しない。あるのは、`1121回` の必死な `mySingleByteIO` ポーリングサイクルと、ハードウェア固有の物理的遅延境界線を正確に踏みしめる「筋肉の記憶」だけだ。

### ハードウェアの幽霊：0xE3 と帯域内シグナリングの罠

時序の乱闘が正面衝突の戦いだとすれば、ハードウェア底層の「幽霊脆弱性」は窒息させるような暗殺劇だった。過酷なストレス環境下で、システムは一定の間隔で不可解な「ハードウェア切断」と「デッドロック」に見舞われた。

幽霊を捕らえるため、私はシステム低層の監視ツールを使い、物理バス上の実際の電気通信流であるIRP（I/Oリクエストパケット）をキャプチャした：

```text
00000152  2016-03-04 13:42:04.9320968  +0.0000062  IRP_MJ_READ  UP  0x00000000  81
00000174  2016-03-04 13:42:12.2557728  +0.0000066  IRP_MJ_READ  UP  0x00000000  cd
```

リターンコードが `81`（正常な `cd` ではない）になったとき、デバイスは完全に沈黙した。数万行に及ぶ16進数の洪水を目視で1バイトずつ剥ぎ取り、午前3時、ついに幽霊の正体を突き止めた。

当時、上位PCは物理オフセットアドレスを含むリードコマンドを送信していた。1つのログエントリは13バイト。12バイトのグローバルヘッダを加え、115番目のログエントリの物理アドレスを算出したとき：
`115 * 13 + 12 = 1507`。16進数に変換すると、それは **`0x05E3`** だった。その下位バイトこそが **`0xE3`** なのだ。

真相は戦慄すべきものだった。まさにこの `0xE3` が、あの独自設計のアダプタの低層ファームウェアプロトコルにおいて、「グローバル・リセット（物理再起動）」コマンドとしてハードコーディングされていたのである！

これは「帯域内シグナリング欺瞞（In-band Signaling Deception）」の最も致命的な形態だ。ホストPCの意図は、アドレスを表す通常の「データ・ペイロード」を送信することにあった。しかし、データチャネルと制御チャネルに厳格な物理的アイソレーション（Out-of-Band隔離）が欠如していたため、ハードウェアアダプタはそれを途中で横取りした。アダプタは、単なるデータバイトである `0xE3` を最高レベルの「制御の権杖（コマンド）」と誤認し、その場で通信を遮断してハードウェア再起動を実行したのだ。

視野を広げてみれば、これこそが今日Web3やクラウド企業を震え上がらせている SQLインジェクション、HTTPリクエストスマグリング、あるいはLog4jのRCE脆弱性と全く同じ、絶対的な根本原因ではないだろうか？データ流と制御流の境界線に対する物理的な敬畏を失ったとき、アーキテクチャの中に幽霊が蔓延る。私の解決策は依然として暴力的で決定的だった。ドライバのカプセル化レイヤーで `0xE3` のアドレス範囲に対して強制的なエスケープと物理的なスキップを実装し、ハードウェアの幽霊を完全に絞め殺したのである。

**今日のハードウェアウォレットメーカーやDePINインフラプロバイダーにとって、厳格なOut-of-bandシグナリング隔離の実装を怠ることは、単なるシステムクラッシュを意味しない。それはリモートでのハードウェアの「文鎮化」や、秘密鍵の完全な流出を意味する。これらの「幽霊」を早期に駆逐することは、企業の存亡に関わる評判の失墜や、壊滅的なリコールキャンペーンから救うことと同義なのだ。**

## 第4章：宿主生命周期の次元的鎮圧 (FreeLibrary & 0xC0000005)

低層DLL開発における最大の苦痛は、自分自身を全く制御不能な「ブラックボックス・ホスト」（例えば、レガシーなIEプロセスが走るATM）の中に押し込めなければならないことだ。一度クラッシュすれば、親切なエラーメッセージなど出ない。Windowsイベントビューアに、冷徹なエラーコード **`0xC0000005`**（アクセス違反）が残るだけだ。

### スタック法医学：.MAPファイルによるアドレス狩り

これは、フロントエンドとバックエンドの境界を越えた呼び出しによる、典型的な領域外アクセスの惨劇だった。現場から送られてきたクラッシュダンプのポインタを受け取った私は、真犯人を見つけ出すためにコンパイラが生成した `.map` メモリマッピングファイルをひっくり返した。ベースアドレスからのオフセットを算出し、犯人を追い詰めるこの「手動計算」こそ、老練なアーキテクトの署名とも言える動作だ。

膨大な命令オフセットの中から、私は実行スタックを強制的に復元し、ホストが崩壊した絶対的な物理アドレスを突き止めた：
```text
App.exe!TUtil::intToString(int val) Line 43 C++
```

この手がかりを辿り、私は `jcIButton.cpp` の `GetTouchKeyLog` コールバックループの末尾へと遡った。

```cpp
// 32ビットモードにおいて、z=7 のときに 0xC0000005 の惨劇が誘発された
for(int z=1; z<=LockLogCount; z++) {
    WriteChar(&DataSend[4], 1); 
    memset(cLogData, 0, sizeof(cLogData));
    ReceiveCharLog(cLogData + 2*i); // 致命的なコールバックのトリガー
}
```

**バイナリレベルの深層分析：**
なぜ、よりによって `z=7` でホストは爆発したのか？原因は、C/C++と外部環境（C#やJava）が対話する際の、極めて危険な**呼び出し規約（Calling Convention）の不一致**にあった。

低層の `ReceiveCharLog` コールバック関数は、DLLコンパイル時にデフォルトの `__cdecl` 規約（呼び出し側がスタックを掃除する）を採用していた。しかし、C#側のホストプログラムはデリゲートを宣言する際、Windows APIの標準である `__stdcall` 規約（呼び出された側がスタックを掃除することを期待する）を使用していたのだ。

このズレにより、`ReceiveCharLog` が呼び出されるたびに、スタックに積まれた引数の領域が正しく解放されずに残った。32ビットx86アーキテクチャでは、ポインタ引数1つにつき4バイトを消費する。1回のループでスタックポインタ（ESP）は4バイト分だけ「漏れ」、ズレていく。7回目のループに達したとき、スタックには28バイトという巨大な亀裂が入っていた。

この28バイトの致命的な滑落が、ベースポインタ（EBP）相対でアドレッシングされていたローカル変数 `z` や `LockLogCount` の物理的な位置を、ゴミデータへと完全に曝露させた。`z++` が実行されたとき、CPUは `z` に巨大な乱数値を割り当てた。結果として、`memset` はその天文学的な数値を「長さ」として受け取り、瞬時に数メガバイトのメモリをゼロで塗りつぶし、ホストプロセスのメモリ空間を完全に破壊して `0xC0000005` を引き起こしたのだ。私は、`[UnmanagedFunctionPointer(CallingConvention.Cdecl)]` による厳格な宣言と `__stdcall` エクスポート制約により、この言語を跨ぐ境界線を完全に溶接した。

### 非同期の暗殺者と FreeLibrary の罠

もう一つの凄惨な教訓は、極限のパフォーマンスへの渇望から得られた。当時、私は `boost::thread` を導入し、1行の書き込みオーバーヘッドがわずか0.4マイクロ秒という、ロックフリーの非同期ログエンジンを作り上げた。テスト環境では完璧に動作したが、銀行の現場ではホストが「音もなく消える」事象が頻発した。

**失敗経路の推論：**
C++ DLLにおいて、ホストからの `FreeLibrary` は「物理的な遮断（断電）」を意味する。ビジネスフローが終了してDLLがアンロードされると、OSのメモリマネージャはDLLのコードセグメント（`.text`）をメモリマップから容赦なく引き剥がす。しかし、Windowsカーネルのスケジューラは、DLL内部で `boost::thread` が密かに生成したバックグラウンドのスレッドを**自動的に殺すことはない**。

当初、私たちはアンロードの前に `Sleep(50)` を入れ、スレッドに猶予を与えようとした。
> *「FreeLibraryの前にSleep(50)を入れるべきだ。書き込みを担当するバックグラウンドスレッドが存在するが、FreeLibraryを呼ぶと即座にそのスレッドの足場が失われる……」*

しかし、これは単なる現実逃避に過ぎなかった。スレッドのスケジューリングは非決定論的であり、I/O待ちで休止していたバックグラウンドスレッドが目を覚ましたとき、実行しようとした次の命令（EIPレジスタが指すアドレス）は、すでに存在しない虚無のメモリへと変わっていたのだ。ページフォールト（Page Fault）が噴出し、プロセスはその場で絶命した。

私はパフォーマンス指標の誘惑を振り切り、アーキテクトとしての最高度の自制心を発揮した。CPUバウンドではない工学的な制御シナリオにおいて、非同期によるマイクロ秒単位の利得は、ライフサイクル管理の破綻という代償に見合わないことを認めたのだ。私はコードを強制同期書き込みへと「次元降下」させ、スレッドの寿命を呼び出し側のスタックフレームに厳格に縛り付けることで、非同期の幽霊を物理的に根絶した。

**エンタープライズ・アーキテクチャにおいて、真の商用価値はピーク時のマイクロベンチマークではなく、99.999%の稼働率とゼロのメンテナンスコストにある。一度もクラッシュしない同期システムは、現場で稀にATMを文鎮化させる爆速の非同期エンジンよりも、無限に高い利益をもたらすのである。**
