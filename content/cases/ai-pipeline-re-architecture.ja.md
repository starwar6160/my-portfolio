# AI パイプライン再設計

## エグゼクティブサマリ

約 500 件のスポーツ分析レコードを処理する本番 AI ワークフローは、すでにビジネス価値を証明していました。

課題は、それを運用可能なスケールにすることでした。

元の実装は次を要していました。

```text
1 回あたり 160-180 分

高い LLM 運用コスト

頻繁なワークフロー障害

低いスケーラビリティ
```

再設計後は次の状態になりました。

```text
1 回あたり約 8 分

16x-22x 高速化

約 99% のコスト削減

高信頼運用
```

この再設計により、プロトタイプ級のワークフローを、本番対応の AI パイプラインへ変換しました。

## 背景

このワークフローは、スポーツのオッズと履歴データを収集し、AI による予測と分析を生成し、MySQL に保存し、下流のレポート配信へつなぐものでした。

小規模では問題なく動作していました。

しかし 1 回あたり約 500 件になると、性能と運用コストが主要制約になりました。

もはや課題は機能追加ではありませんでした。

課題は、スケールしても持続可能に運用できることでした。

## 課題

このワークフローには複数のアーキテクチャ上のボトルネックがありました。

```text
N+1 の AI リクエスト

過大なペイロード

状態喪失

DB 障害

レート制限圧

認証情報の脆弱性
```

ハードウェアやモデルを増強しても、拡張性は十分に改善しませんでした。

## 根本原因分析

問題はモデル品質ではありませんでした。

アーキテクチャでした。

ワークフローは各層でレイテンシを増幅していました。

```text
データ
↓
AI
↓
データベース
↓
レポート
```

各ステージで不要なシリアライズ、同期、ネットワークオーバーヘッドが発生していました。

実行パターンは次のようなものでした。

```text
1 レコード
↓
1 回の AI リクエスト
↓
1 回の DB 操作
↓
これを 500 回繰り返す
```

この構造では、モデル性能に関係なくスループットは悪化します。

ボトルネックはデータフローそのものにありました。

## 実施内容

### スループット指向の処理

#### バッチ指向 AI アーキテクチャ

リクエスト単位の処理を、スループット指向の処理へ置き換えました。

個別処理ではなく、約 20〜25 件を 1 バッチにまとめて AI パイプラインへ流すようにしました。

効果:

```text
ネットワーク往復 約95%削減

AI 利用率向上

レート制限露出の低減

パイプライン遅延の削減
```

これにより、ワークフローは request-oriented から throughput-oriented へ変わりました。

### ペイロード設計

#### 入力の極小化

元の上流ペイロードは 1 件あたりおよそ:

```text
約 8KB
```

その大半は分析価値が低く、トークン消費だけを増やしていました。

専用 ETL レイヤで次だけを残しました。

* チーム情報
* リーグメタデータ
* 試合時刻
* コアオッズデータ

結果:

```text
8KB
↓
約 500B

約 93% の削減
```

効果:

* トークン消費の削減
* 推論効率の改善
* メモリ圧力の低減
* 出力の一貫性向上

### コストを意識した AI アーキテクチャ

#### ワークロードに応じたモデルルーティング

すべてのリクエストに高価な単一モデルを使うのではなく、タスクの複雑さに応じてモデルを切り替えるようにしました。

構成:

```text
標準ケース
↓
高スループットモデル

複雑ケース
↓
高度推論モデル
```

効果:

```text
運用コスト低減

スループット向上

リソース利用の最適化
```

これにより、モデル選定を convenience ではなく economics の観点に切り替えました。

### データ基盤設計

#### データリネージの保全

最も重大な失敗のひとつは、AI 処理後にメタデータが失われることでした。

予測は生成できても、次のような識別子が下流処理で失われることがありました。

```text
match_id
```

これを防ぐために:

```text
AI 処理ブランチ
+
メタデータブランチ
↓
決定論的マージ
```

を導入しました。

これにより次を維持できました。

```text
Single Source of Truth

Data Lineage

Referential Integrity
```

### 信頼性工学

#### Dead-Letter Queue パターン

本番では、無効入力があることを前提にすべきです。

スキーマ変更、NULL 値、不正レコードが原因で、以前はワークフロー失敗や DB トランザクションエラーが発生していました。

次を実装しました。

```text
検証
↓
Dead-Letter Path
↓
Telegram 通知
```

不正レコードは、正常処理を止めずに隔離されるようになりました。

効果:

```text
障害の分離

運用可視性

継続処理
```

追加の強化として:

* 永続的なワークフロー状態
* 明示的な暗号鍵管理
* コンテナ再起動後も維持される認証情報

## 成果

| 指標 | Before | After |
| --- | ---: | ---: |
| 処理時間 | 160-180 分 | 約 8 分 |
| 高速化 | - | 16x-22x |
| LLM コスト | 約 $15-20/日相当 | 約 $0.13/回 |
| ペイロードサイズ | 約 8KB | 約 500B |
| 削減率 | - | 約 93% |
| API パターン | レコード単位 | バッチ処理 |
| 障害処理 | パイプライン中断 | DLQ + 通知 |
| スケーラビリティ | 限定的 | 本番対応 |

## 戦略的成果

再設計により、主要ボトルネックは AI 推論コストから、ビジネス側のスループット要件へと移りました。

これにより、インフラ費用、API 費用、運用複雑性を比例的に増やさずに、持続可能なスケールが可能になりました。

## この事例の意味

多くの AI ワークフロー障害は、モデル品質ではなくアーキテクチャが原因です。

この案件は次を示しました。

```text
Throughput

Cost

Reliability

Maintainability
```

は、モデル能力ではなくデータフロー設計で決まることが多いということです。

同じアーキテクチャパターンは次へ直接応用できます。

* AI Platform Engineering
* LLM Infrastructure
* Agent Orchestration
* Workflow Automation
* Data Platforms
* AI Operations

この仕事は、成功したプロトタイプを、アーキテクチャ、コスト設計、信頼性設計で本番スケールへ変換する能力を示しました。
