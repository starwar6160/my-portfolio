# 「言語の壁」で死ぬシステム ― C++ × C#/Java 跨言語インターフェースの防崩設計 5 原則

:::message
この記事は、金融端末・暗号SDK・組み込みセキュリティ機器の現場で10年以上にわたり蓄積してきた「言語間インターフェース障害」の実戦記録をもとに、設計原則と具体的な対処法を体系化したものです。
:::

## はじめに ― なぜ「ブリッジ層」が最も壊れやすいのか

現代のエンタープライズシステムは、ほぼ例外なくポリグロット（多言語混成）です。Java がビジネスロジックを担い、C# が UI を描画し、Python がスクリプトを書き、すると ― C++ が暗号計算・ハードウェアドライバ・高性能アルゴリズムといった「重量物の持ち上げ」を担当します。

業界はこの多様性を讃えます。マイクロサービス、ポリグロット・パーシステンス、「適材適所」。

しかし、業界がほとんど **認めようとしない** 事実があります。**言語間の境界面は、システム全体で最も脆い耐荷重構造である** ということです。ここではメモリモデルが衝突し、呼び出し規約がスタックを静かに破壊し、ガベージコレクタが確定的なリソース管理と戦争を繰り広げ、たった一つのキャッチされない例外がホストプロセス全体を消滅させます。

筆者は、銀行 ATM 端末システム、組み込みハードウェアセキュリティモジュール（HSM）、国家標準暗号 SDK（SM9/IBC）、Internet Explorer 上で動作する ActiveX コントロール（Windows XP 環境、本番稼働中）など、**失敗が許容されない環境** でこれらの言語間ブリッジの構築・強化に携わってきました。

以下、5 つの防崩メカニズムを、具体的な障害事例・コード・性能データとともに解説いたします。

## メカニズム 1：データ型の「降維」伝達 ― ヒープ破損の根絶 {#mechanism-1}

### 障害シナリオ

最も一般的かつ最も壊滅的な言語間障害モード ― **Heap Corruption（ヒープ破損）**。

銀行端末プロジェクトで、関数 `zwGetJcxmlMsgType` が本番クラッシュを引き起こしました。この関数は内部で `boost::property_tree` を使用して XML を解析しています。DLL 自体のテストハーネスでは問題なく動作しますが、外部の MFC アプリケーションから呼び出した瞬間にクラッシュします。

:::details 根本原因
DLL は VC2010 の CRT でコンパイル。MFC アプリケーションは VC6 の CRT でコンパイル。**2 つの異なるアロケータ**が同一オブジェクトに対して `alloc` / `free` を実行し、ヒープメタデータが破壊されました。`RtlValidateHeap` がオフセット `B688` で一貫してエラーを報告しています。
:::

### 設計原則：インターフェースのプリミティブ化

**DLL 境界を越えて複雑な C++ オブジェクトを渡してはいけません。** `std::string` も、`boost::property_tree` も、STL コンテナも一切禁止です。

```cpp
// NG：DLL 境界を越える複雑な C++ オブジェクト
DLL_API boost::property_tree::ptree ParseConfig(const std::string& xml);

// OK：C スタイルのプリミティブに降維
DLL_API int ParseConfig(const char* xmlInput, 
                         char* jsonOutput, 
                         int outputBufferSize);
```

**複雑な構造化データの場合：**

Google Protocol Buffers（Protobuf）を **言語間シリアライゼーション基盤** として採用しました。

| 言語側 | API | オーバーヘッド |
|---|---|---|
| C++ | `SerializeToOstream()` | マイクロ秒レベル |
| Java | `parseFrom(byte[])` | 2 行のコードで完全復元 |

:::message alert
**バイナリデータの `\0` 切り詰め問題：** 暗号署名データ（例：296 バイト）を C 文字列として JNI 経由で渡すと、途中の `0x00` バイトで切り詰められます。SWIG の `%apply (char *STRING, size_t LENGTH)` マッピングで `byte[]` に変換し、バイナリ完全性を保証してください。
:::

### Mechanism vs. Policy 分離

SM9 暗号 SDK の旧版 `zxtibc.dll` は「三無コード」（ドキュメントなし・コメントなし・単体テストなし）でした。さらに、内部で `KGSName` と `Version` からディスクパスを計算して PEM 証書を読み込む設計 ― **環境依存度が極めて高い設計** でした。`readIbCertFromLocalFile` は恒常的にエラーコード `-6` を返却していました。

リファクタリング後の `zxtibcLite` は、PEM 内容をメモリ上の Base64 エンコードバッファとして受け取ります。ライブラリは**暗号計算のみ**を担当する純粋な計算エンジンとなりました。

> **「環境パスが制御不可能」→「上位層がストレージ読み取りを担当」→「ライブラリはメモリデータの復号・検証のみ担当」**

この「機制と策略の分離」（Mechanism vs. Policy）は、OS 設計から借用した概念であり、クラウドネイティブ/コンテナ化への適応力を本質的に高めます。

## メカニズム 2：呼び出し規約の強制整合 ― ESP スタック破壊の防止 {#mechanism-2}

### 障害シナリオ

C# アプリケーションが C++ DLL の `SwapTouchKeyLog` を呼び出すと、`0xC0000005`（Access Violation）でクラッシュ。`OutputDebugString` によるトレースでは関数内部のロジックは正常に完了・復帰しています。**クラッシュは関数リターン後に発生**。

### 根本原因分析

| | C++ 側 | C#/.NET 側 |
|---|---|---|
| デフォルト規約 | `__cdecl`（呼び出し元がスタック清掃） | `__stdcall`（呼び出し先がスタック清掃） |

C++ 関数が `__cdecl` で戻ると ESP を調整しません。C# 側は `__stdcall` を前提として自身でもスタックを調整します。ESP レジスタが二重操作により破壊され、次のメモリアクセスでクラッシュします。

### 修正：明示的規約宣言 + .DEF ファイルによるシンボル固定

```cpp
// コールバックプロトタイプ ― C# デリゲートの規約に合わせる
typedef void (__stdcall *ReturnLockLog)(char* Item, 
                                        char* EventTime, 
                                        char* Factor);
```

:::message
**名前修飾（Name Mangling）の罠：** `__stdcall` を追加すると、C++ コンパイラがエクスポートシンボル名を装飾します（例：`_SwapTouchKeyLog@4`）。Java や C# は装飾なしの名前を探すため「エントリポイントが見つかりません」エラーになります。**.DEF ファイル** でシンボル名を明示的に固定してください。
:::

```
; .DEF ファイル ― Name Mangling 回避
EXPORTS
    Open                @1
    Close               @2
    Notify              @3
    SetRecvMsgRotine    @4
    SwapTouchKeyLog     @5
```

### マルチベンダー環境での規約統一

SM9 暗号 SDK プロジェクトでは、当初すべてを `__stdcall` に統一する計画でした。しかし、OpenSSL のプリコンパイルライブラリは `__cdecl` を使用しており、OpenSSL の再ビルドは不可能でした。

**解決：`__cdecl` に統一的に後退し**、呼び出し側（C# デリゲート、Java JNA マッピング）で規約差分を吸収しました。18 個の未解決外部シンボルの修正が必要でしたが、ABI の一貫性を確保できました。

> **一貫性は規約に勝る。** どの規約を選択するかは重要ではありません ― コールチェーンの全参加者が同意していることが重要です。

## メカニズム 3：確定的ライフサイクル回収 ― GC からハードウェアハンドルを奪還 {#mechanism-3}

### 障害シナリオ

HID ハードウェアセキュリティモジュール（密盒）プロジェクトにおける障害パターン：

1. C# ビジネスロジックが `JcSecBox` ラッパーをインスタンス化
2. コンストラクタが `hid_open()` を呼び出し（USB HID デバイスの排他アクセス取得）
3. ビジネスロジック完了。ラッパーがスコープ外に
4. **GC がオブジェクトを即座に finalize しない。**`hid_close()` を含むデストラクタが呼び出されない
5. ループの次のイテレーションで新しい `JcSecBox` が `hid_open()` を試行
6. デバイスは前のオブジェクト（未 finalize）に占有されたまま → **ポートデッドロック**

ストレステスト（1 リクエスト/秒）で、約 130-140 回の操作後に `0xC0000005` クラッシュが安定的に発生。

**比較実験結果：**

| モード | 連続実行回数 | ログサイズ | クラッシュ |
|---|---|---|---|
| C#（GC 依存） | ~130 回 | 4.6 MB | あり |
| C++（Direct） | 1,513 回（4,500+ 認証） | 2.7 MB | **なし** |

### 対策：明示的・ステートレス・ライフサイクル管理

デストラクタ/ファイナライザからの**自動リソース解放を全面撤去**。各操作を完全ステートレスに設計：

```
Open → Authenticate → [Business Logic] → Close
```

```csharp
// OK：明示的・確定的・交渉不可
var box = new JcSecBox();
try {
    box.Open();
    box.Authenticate();
    var data = box.Read();
} finally {
    box.Close();  // 確定的。即座。非交渉。
}
```

### パフォーマンス計測（QueryPerformanceCounter, ~2.7MHz, 精度 0.1ms）

| 操作 | 合計時間 | 実 I/O | ログ・オーバーヘッド |
|---|---|---|---|
| デバイス Open | 10 ms | 10 ms | ― |
| 認証 | 70 ms | 10 ms | 60 ms |
| 書き込み（認証込み） | 150 ms | 60 ms | 90 ms |
| 読み取り（認証込み） | 200 ms | 70 ms | 130 ms |

:::message alert
**Poco ログライブラリの衝撃：** 1 エントリあたり 20 ms。`printf` + `OutputDebugString` は合計わずか 0.4 ms。高頻度暗号演算において、**ログ I/O が最大のパフォーマンスボトルネック**です。本番環境では設定ファイルによるログレベル動的切り替えを実装し、ログローテーション（45 MB → 1.5 MB `.gz` セグメント）でストレージ圧力を管理しました。
:::

## メカニズム 4：例外の隔離室 ― ホストコンテナの保護 {#mechanism-4}

### 障害シナリオ

ATM 端末プロジェクトでの DLL ロードチェーン：

```
IE ブラウザ → サードパーティ OCX → 当方 DLL → 錠前ハードウェア（WebSocket）
```

物理 USB ケーブルが取引中に切断されると、バックグラウンド通信スレッドの `receive()` が `Poco::Exception` を送出。この例外がスレッド関数内でキャッチされず、OCX コンテナ境界を越えて IE のプロセス空間に到達。**IE がフリーズし、ATM オペレータがハードリブートを余儀なくされました。**

さらに危険な障害モード：**グローバル静的オブジェクトの初期化**。DLL がネットワーク接続マネージャをグローバル静態として宣言。コンストラクタが `DllMain` 中で WebSocket 接続を試行。ハードウェアがオフラインの場合、**DLL ロード時に例外が発生**し、IE が回復不能状態に陥りました。

### 対策：例外の完全封じ込め

**原則：C++ 例外は DLL 境界を絶対に越えてはならない。**

```cpp
void BackgroundCommThread() {
    try {
        wsClient->connect();
        while (running) {
            try {
                auto msg = wsClient->receive();
                processMessage(msg);
            } catch (Poco::Exception& e) {
                OutputDebugStringA(e.displayText().c_str());
                // 内部復旧：サイレント再接続
            } catch (...) {
                OutputDebugStringA("Unknown receive failure");
            }
        }
    } catch (...) {
        OutputDebugStringA("Unknown network failure in thread");
    }
    // スレッドは正常終了。ホストプロセスは無傷。
}
```

### RAII による最終安全ネット

接続オブジェクトをスレッド関数内の**ローカルスタック変数**として宣言。例外による `return` 時に C++ のスタック巻き戻し（Stack Unwinding）が自動的にデストラクタを呼び出し、無効なソケットハンドルを解放します。

**検証：** ホットプラグテスト ― 通信中に USB ケーブルを物理的に抜き差し（約 20 秒遅延）― システムは次の `Notify()` 呼び出しでサイレントに接続を再確立。ホスト側に異常は一切発生しませんでした。

### エラーコード階層化

底層 DLL のエラーコード（`0-5`）との衝突を避けるため、ミドルウェアエラーコードは `16` から開始：

| コード | 意味 |
|---|---|
| 16 | `JCELOCK_ERROR_DLL_NOTFOUND` |
| 17 | `JCELOCK_ERROR_DLLFUNC_NOTFOUND` |
| 18 | `JCELOCK_ERROR_INPUTMSG_NULL` |

## メカニズム 5：型昇格とビット幅整合 ― タイムトラベル・バグの殲滅 {#mechanism-5}

### 障害シナリオ：30 年の時間巻き戻し

SM9 IBC 暗号 SDK の JNI 統合で発見されたデータ破損：

| 側 | 値 | 意味 |
|---|---|---|
| C++ | `1375891200` | 2013-08-07 16:00:00（北京時間） |
| Java | `424567760` | 約 1983 年 |

証書の有効期限タイムスタンプが **30 年巻き戻り**、証書検証がエラーコード `-6` で失敗。例外もクラッシュも警告もなく、**サイレント**に発生。

### 根本原因分析

`time_t` は C++ では通常 `long`（32 ビットプラットフォームでは 32 ビット）。Java の `long` は常に 64 ビット。SWIG のデフォルトマッピングが上位 32 ビットを切り詰め。

### 修正：SWIG Typemap の明示的強制

```swig
// time_t を Java の 64 ビット long に強制マッピング
%typemap(jni) time_t "jlong"
%typemap(jtype) time_t "long"
%typemap(jstype) time_t "long"
```

### JNA による代替アプローチ

パラメータ型がシンプルなインターフェースには、**JNA (Java Native Access)** を JNI の代替として採用しました。JNA はランタイムリフレクションにより C 関数シグネチャを動的にマッピングし、`_wrap.c` グルーコード・SWIG インターフェースファイル・関連するポインタ演算とメモリリーク・リスクを**完全に排除**します。

**検証：** `CrlContentToFile` を JNA 経由でラッパーコードなしに呼び出し成功。シンプルな API サーフェスに対するアプローチの信頼性を実証しました。

:::message
**トレードオフ：** JNA のリフレクションベースの動的ディスパッチは、高頻度署名シナリオで許容できないオーバーヘッドを生じる可能性があります。本番ボリュームの署名ワークロードで JNA + Protobuf と ネイティブ JNI をベンチマーク比較した上で、完全移行を決定することを推奨します。
:::

## 補論 A：クロスコンパイラ時空 ― VC6 環境への現代 C++ 注入 {#appendix-a}

銀行 ATM 端末は Windows XP + IE6。ホストアプリケーションは VC6（1998 年リリース）でビルド。統合する DLL は VS2010 + Boost 1.56 + Poco 1.4.6 + MIRACL。

| 試行 | 方法 | 結果 |
|---|---|---|
| 1 | VC2010 再頒布パッケージ（4 MB）インストール | VC6/VS2003 CRT と衝突、再起動異常 |
| 2 | `MSVCP100.DLL`, `MSVCR100.DLL` を手動コピー | スタンドアロン XP VM で OK、OCX 経由で -1 |
| 3 | **全静的リンク（`/MT`）** | **931 KB の単一 DLL、外部依存ゼロ。成功** |

**結論：** ホストランタイムを制御できない環境では、**静的リンクは最適化ではなく生存要件**です。これは、ハードウェアウォレット・ファームウェア、エアギャップ署名機、あらゆるセキュリティクリティカルな組み込みシステムと同じ原則です ― **完全独立による実行確定性**。

## 補論 B：RPC 進化 ― DLL から WebSocket サービスへ {#appendix-b}

### 技術選定経緯

| 技術 | 判定 | 理由 |
|---|---|---|
| ZeroC ICE | 評価・棄却 | `ice.jar` 依存が重い；Java クライアントでランタイム例外 |
| ZeroMQ | 評価・棄却 | `bind()` がウイルス対策ソフトに静かにインターセプトされる；`jzmq` のコンパイル複雑度 |
| WebSocket + JSON (Poco) | **採用** | 自然な言語間互換性；錠前ハードウェアとプロトコル再利用 |

### 並行ストレステスト結果

- **構成：** `boost::thread` 100 並行 → シングルスレッド WebSocket サーバー
- **Linux vs. Windows (ICE 3.3.1)：** 100 並行では性能同等。1,000 超で Windows スループット崩壊。Linux は 3,000 並行で応答時間 20 ms → 39 ms（CPU 1 コア未飽和）

### SM9 RPC プロトコル設計

「双パケット・スニフ」ルーティング：クライアントがまず `func_type` 列挙型を送信、サーバーがハンドラにルーティング。その後、実データペイロードを第 2 メッセージで送信。レスポンスはリクエスト構造を再利用し、OUT フィールドを埋めて返却。

```protobuf
message sm9_function_name {
    int32 func_type = 1;  // 5=SetUp, 6=Extract, 7=Encrypt, 
                           // 8=Decrypt, 9=Sign, 10=Verify
}
```

## 設計哲学：リソース・リスペクト {#philosophy}

この分野で 10 年以上の実務を経て、ひとつの指導原則に集約しました ― **リソース・リスペクト (Resource Respect)**。

リソース ― メモリバッファ、ハードウェアハンドル、ファイルディスクリプタ、スレッドコンテキスト ― にはライフサイクルがあります。単一言語・単一ランタイムのシステムでは、言語自身のメカニズム（RAII、GC、try-with-resources）がこのライフサイクルを適切に管理します。

言語間境界では、**誰も責任者がいません**。C++ アロケータは C# のガベージコレクタを知りません。Java のファイナライザは HID デバイスの排他アクセス制約を知りません。ActiveX コンテナは Poco WebSocket の例外モデルを知りません。

**リソース・リスペクトとは：**

1. **すべてのリソースには唯一のオーナーがある。** オーナーシップは言語境界を越えない
2. **取得と解放は明示的。** 暗黙の構築なし、遅延ファイナライゼーションなし
3. **エラーは封じ込める。** 例外は境界でキャッチ。エラーコードを返却。ホストプロセスを決して危険にさらさない
4. **データは単純化する。** 複雑なオブジェクトは境界を越える前にプリミティブストリームにシリアライズ。ワイヤフォーマットが契約
5. **インターフェースはファイアウォール。** 双方のランタイム前提から互いを保護する

システムがますますポリグロット化し、AI 生成コードが既存スタックに新しい言語を導入し、クラウドネイティブアーキテクチャがモノリスを通信サービスのメッシュに分解する今日、インターフェース境界はもはやエッジケースではありません。それは現代ソフトウェアの **中枢神経系統** です。

強化するか、さもなくば強化されるか。
