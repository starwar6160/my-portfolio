# Merge-Sport: Spring Boot データ取り込み高速化

## プロジェクト概要

Spring Boot 3 / Java 17 で構築されたスポーツデータ統合基盤で、外部の試合データを取得して下流サービスへ同期していました。

トラフィックが増えるにつれて取り込みパイプラインが主要ボトルネックになり、HTTP ポーリング、1件ずつの永続化、そして結合度の高い実装が、スループット低下・ロック競合・運用リスクの増加を招いていました。

目的は単なる高速化ではなく、今後の成長に耐えられるように、同期パイプラインをスケーラブル・高信頼・保守しやすい構造へ変えることでした。

## 課題

### DB ラウンドトリップの過多

元の実装はレコードを個別に処理していました。

500件の同期処理に 70 秒以上かかり、その大半がデータベースとの反復的なやり取りに費やされていました。

### N+1 的な永続化パターン

重複排除と更新処理が 1件ずつ DB を参照・更新しており、不要な遅延とロック競合を生んでいました。

### スケーラビリティ不足

識別キーの重要な列に適切な索引がなく、フルテーブルスキャンと高負荷時のデッドロックリスクが増えていました。

### 巨大化した処理ロジック

10以上の外部提供元を、洗浄・変換・マージ・永続化が混在した 1200 行超の単一サービスで処理していました。

## アーキテクチャ改善

### 1. バッチ UPSERT エンジン

1件ずつの永続化をやめ、MyBatis のバッチ処理と MySQL の UPSERT パターンに切り替えました。

* `INSERT ... ON DUPLICATE KEY UPDATE` を使用
* 多数の書き込みを 1 回のトランザクションに集約
* ネットワークと DB のオーバーヘッドを大幅に削減

### 2. 索引設計の見直し

重複排除の重要経路をプロファイリングし、識別用の複合ユニーク索引を追加しました。

結果:

* フルテーブルスキャンを解消
* ロック保持時間を短縮
* 同時更新時の安定性を改善

### 3. Strategy Pattern による分離

ソースごとに異なる処理を Strategy Pattern で分離しました。

結果:

* 1200 行超のモノリスを分解
* 保守性とテスト容易性を改善
* 新しいデータ提供元の追加を簡素化

### 4. 信頼性と可観測性の強化

本番運用に耐える診断と検証を追加しました。

* MDC によるエンドツーエンドのリクエスト追跡
* ログストームを避けるためのサンプリング出力
* Testcontainers を使った統合テスト
* テスト起動時のスケジュールジョブ分離

### 5. 通信方式の見直し

必要な箇所では、反復的な HTTP ポーリングをやめ、WebSocket や MQTT を使った長寿命の push 型通信へ移行しました。

これにより、提供元側のレート制限の影響を受けにくくなり、データ鮮度も改善しました。

## 成果

### パフォーマンス

* 500件同期を約 70 秒から 1.5 秒へ短縮

### スケーラビリティ

* 水平スケールなしで約 10 倍の同期量増加に対応

### 信頼性

* デッドロック発生確率を低減
* 同時更新時の挙動を改善
* 同期結果の予測可能性を向上

### 保守性

* 結合度の高いモノリスをモジュール化
* テストと将来の機能拡張を容易化

## 事業インパクト

データ鮮度の改善、運用リスクの低減、下流利用者への提供予測性の向上を実現しました。

アーキテクチャ上のボトルネックを解消したことで、単なるインフラ増強に頼らず、今後の成長を吸収しやすい基盤になりました。
