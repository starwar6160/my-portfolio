# 本番圧力下でのフロントエンド安定化

## 背景

2025 年後半、前任のフロントエンド担当が離任した後、問題の多い Vue 3 + TypeScript 製 IM フロントエンドを一時的に引き継ぎました。

システムには、業務側のテストを止めるレベルの P0/P1 問題が複数ありました。

- リフレッシュ時に強制ログアウトが発生
- WebSocket 接続が不安定
- ローディングやリトライ遷移で UI がデッドロック
- 通常利用でも白画面がランダムに出る
- Cypress の自動テストが不安定
- 認証状態と非同期 UI 更新が競合
- フロントエンド状態とバックエンド契約がずれる
- Int64 / Snowflake ID が JavaScript の精度制約に触れる

## 制約

私はこの案件の専任フロントエンド担当ではありませんでした。

また、コードベース全体を手でじっくり読み込む時間もありませんでした。本番圧力のある状況で、限られた文脈から素早く安定化する必要がありました。

## 調査方法

大規模なリファクタリングから始めず、実行時の事実から切り分けました。

- 本番ログとエラートレース
- ブラウザ上の再現挙動
- ネットワークタイミングとリクエスト順序
- クライアント側の state transition
- Cypress の失敗点

フロントエンドを、バックエンド障害対応と同じ考え方で扱いました。

- 失敗を再現する
- 状態遷移の境界を特定する
- 実行時に契約を検証する
- 小さく隔離されたパッチを当てる
- 同じ経路を再テストする

## 実装方針

AI は「コード分析と実装の加速器」として使い、真実のソースにはしませんでした。

実際の流れは次の通りです。

1. ログ、トレース、実行時挙動を先に確認する
2. 複数の LLM に対して、失敗モードを踏まえた指示を与える
3. 提案パッチを観測結果と照合してレビューする
4. 原子的な小パッチとして適用する
5. 自動テストとブラウザ確認で再検証する

失敗要因が複数層で絡んでいたため、広い範囲の書き換えは逆に不安定化を招くと判断しました。

### 証拠駆動の進め方

このフロントエンド安定化は、基本的に evidence-driven で進めました。ページを手で延々とクリックしたり、コードベース全体を行単位で読み込んだりする方法には頼っていません。Cypress で主要ユーザーフローを確認し、バックエンド API の統合テストでサーバ側の契約を検証し、ログ・実行時挙動・スタックトレース・ネットワークタイミングから失敗境界を特定しました。その上で、AI に小さな原子的パッチを生成させ、同じテストを再実行して各修正を確認しました。

これは従来の「手で画面を見ながら直す」やり方より、フロントエンドのインシデント対応に近い進め方です。各失敗を再現可能にし、修正範囲を狭く保ち、回帰を見える化することを目的にしました。

## 安定化した領域

### 認証とセッション

- リフレッシュ時の強制ログアウト経路を修正
- 認証状態と非同期 UI 初期化を同期
- トークン更新、ルートガード、メモリ上のセッション状態の競合を解消

### WebSocket ライフサイクル

- 接続ライフサイクルを整理
- 古い socket を明示的に close
- 多重購読と不整合な再接続挙動を防止

### UI 状態の安定化

- ローディングとリトライ遷移を安定化
- 長時間画面のデッドロック条件を除去
- 非同期 state 更新を締め直し、表示と要求状態のずれを抑制

### 契約検証

- 不安全な payload に対する runtime schema check を追加
- UI ローカルの前提に頼らず、境界でデータを正規化
- Snowflake 対応の ID normalization layer を導入

### テスト安定化

- タイミング依存の強い画面に対して Cypress を安定化
- 失敗しやすいロジック経路の Vitest を整備
- 各修正後の regression lock として失敗テストを残す

## アーキテクチャ上の要点

問題の本質は単一バグではなく、複数層の不整合でした。

- 認証状態が single source of truth になっていない
- WebSocket ライフサイクルが決定論的でない
- 非同期 UI 遷移が契約境界で保護されていない
- 数値 ID がバックエンド形式に対して緩く扱われている

対処は backend-style の reliability thinking で進めました。

- 境界を定義する
- エッジでデータを正規化する
- state transition を明示する
- 小さいパッチで速く検証できる形に保つ

## 結果

- 数日で 70 件超の critical issue を安定化
- フロントエンドテストの再現性が大きく向上
- ランダム失敗が大幅に減少
- QA と業務側検証を進められる程度まで挙動が予測可能に
- ビジネス側テストの主要ブロッカーを除去
