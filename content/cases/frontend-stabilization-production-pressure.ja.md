# 本番圧力下のフロントエンド安定化

## エグゼクティブサマリ

Vue 3 / TypeScript のリアルタイムメッセージング基盤は、フロントエンドの引き継ぎ後に本番クリティカルな状態へ入りました。

数日間で次を達成しました。

```text
74+ 件の重大問題を安定化

120+ 件の原子的な本番修正を投入

認証失敗を解消

WebSocket の信頼性を回復

事業側 QA を再開可能にした
```

復旧作業では、認証、WebSocket ライフサイクル、フロントエンドの状態遷移、バックエンド契約境界にまたがる、体系的な状態同期失敗を特定・解消することに注力しました。

大規模な書き換えではなく、証拠ベースのデバッグ、実行時分析、決定論的な所有ルール、小さく低リスクな修正で安定化しました。

## 背景

この基盤は次を支えていました。

* リアルタイムメッセージング
* WebSocket 通信
* 認証・認可
* ブラウザベースの業務フロー
* 自動 QA 検証

P0/P1 の問題が増え、テスト、納品、顧客検証を妨げていました。

見えていた症状は次の通りです。

* リフレッシュ時の強制ログアウト
* WebSocket の不安定化
* 白画面
* UI のデッドロック
* 認証レース
* Cypress の不安定化
* 契約不一致
* Snowflake ID の精度リスク

## 課題

認証、メッセージング、ルーティング、UI 状態管理にまたがって、複数の失敗モードが相互作用していました。

障害同士が重なり、症状が互いを隠すため、個別修正が別の障害を生みやすい状態でした。

主な課題は、本番安定性を維持しながら失敗境界を切り分けることでした。

制約は次の通りです。

* 引き継いだ大きなコードベース
* 限られた立ち上がり時間
* 継続する納期圧力
* 同時多発する P0/P1 障害

安定化には、追加の回帰を出さずに回復することが必要でした。

## 調査方針

フロントエンドを UI プロジェクトではなく、本番障害として扱いました。

調査では実行時の証拠を重視しました。

* 本番ログ
* ブラウザ挙動
* ネットワークタイミング
* 状態遷移トレース
* 認証フロー
* Cypress の失敗境界

復旧フローは次の通りでした。

```text
再現
↓
観測
↓
失敗境界の特定
↓
原子的パッチの適用
↓
検証
↓
反復
```

この進め方により、回帰リスクを抑えつつ迅速に回復できました。

## 根本原因分析

### 所有権の不明確さ

多くの障害は、状態遷移の所有権が曖昧だったことから発生していました。

* 認証
* WebSocket ライフサイクル
* UI 状態
* バックエンド契約

このため、複数サブシステム間で状態の乖離が蓄積していました。

見かけ上は別々の問題に見えました。

本質は整合性でした。

### 認証の所有権

認証状態が同時に次へ存在していました。

* ルートガード
* リフレッシュフロー
* メモリ状態
* トークンライフサイクル

これらはリフレッシュや再試行で乖離し得ました。

結果:

```text
セッション喪失

強制ログアウト

ナビゲーション破綻
```

### イベントライフサイクルの所有権

WebSocket 接続の所有権が決定論的ではありませんでした。

観測された挙動:

* 重複サブスクリプション
* 古いソケット
* 競合する再接続ロジック

結果:

```text
メッセージ欠落

状態不整合

予測不能な UX
```

### 契約の所有権

フロントエンドの前提がバックエンドのペイロード契約からずれていました。

追加のリスクは次の領域にもありました。

```text
Snowflake ID

Int64 値
```

JavaScript の精度境界により、微妙な本番障害を引き起こし得ました。

## 実施内容

### 決定論的な認証

* 認証所有権を統一
* リフレッシュ起因のログアウト経路を削除
* トークンライフサイクルのレースを排除

### イベントライフサイクルの安定化

* WebSocket ライフサイクルを明示化
* 古い接続参照を削除
* 決定論的な再接続動作を実装

### 実行時契約の強制

* API 境界でスキーマ検証
* ペイロード正規化
* Snowflake 安全な ID 処理

### テスト基盤の復旧

* Cypress 実行経路を安定化
* 失敗を回帰ロックへ変換
* 自動検証カバレッジを改善

### デリバリー戦略

* 小さな原子的コミット
* 各パッチ後の実行時検証
* ロールアウト前の失敗経路検証

これにより、影響範囲を小さく保ちながら回復を加速できました。

## この事例の意味

表面上はフロントエンドの障害に見えても、根本問題は分散システム型の整合性失敗でした。

* 認証の所有権
* 契約同期
* 状態遷移
* イベントライフサイクル管理

復旧方法は次の通りです。

```text
失敗境界の特定
↓
決定論的な所有権
↓
実行時検証
↓
段階的な修復
```

この方法は次に直接適用できます。

* バックエンド基盤
* クラウドサービス
* 分散システム
* データ基盤
* AI インフラ

この案件は、未知の本番システムの所有権を素早く引き継ぎ、体系的な失敗を切り分け、強い運用圧力の下で提供能力を回復できることを示しました。
