# Toyota OMRM: ハイブリッドクラウドのリアルタイム AI プラットフォーム

## 営業・採用担当向け要約

- AWS + Azure のハイブリッド AI プラットフォーム事例です。
- リアルタイム音声解析、話者属性推定、クロスクラウド整合を扱いました。
- 主な改善は、WebSocket ストリーミング、イベント駆動処理、時間軸整合です。
- AI Platform、リアルタイム処理、クラウド連携、イベント駆動案件に適しています。
- 詳細な技術説明は以下をご確認ください。

## エグゼクティブサマリ

大手自動車メーカー向けに、リアルタイムの AI 支援型セールスコーチング基盤を設計・実装しました。

このプラットフォームは、顧客との会話をリアルタイム解析し、通話中のレコメンドを生成し、さらに通話後の学習・QA・運用分析向けの構造化レコードを生成しました。

主な機能は次の通りです。

```text
リアルタイム音声インテリジェンス

AWS + Azure のハイブリッド AI パイプライン

話者属性の特定

意図抽出

イベント駆動処理

サーバーレススケーリング
```

本質的な課題は音声認識そのものではありませんでした。

複数の AI システム、ストリーミング基盤、分散処理パイプラインを、ひとつのリアルタイム基盤として整合させることでした。

## アーキテクチャ概要

```text
音声ストリーム
↓
WebSocket ゲートウェイ
↓
サーバーレス・ストリーミング・パイプライン
↓
Azure Speech + CLU
↓
AWS Transcribe
↓
タイムライン整合エンジン
↓
会話インテリジェンス
↓
リアルタイム・コーチング
```

このシステムは次を統合していました。

* AWS Lambda
* API Gateway WebSocket
* EventBridge
* S3
* RDS MySQL
* Azure Speech Services
* Azure CLU
* AWS Transcribe

それらを単一のイベント駆動アーキテクチャにまとめました。

## 課題 #1

### ステートレス基盤上でのリアルタイム音声処理

入力音声は次の形式でした。

```text
WebM / Opus
```

一方で、下流の AI サービスは次を期待していました。

```text
PCM / WAV
```

処理はすべて AWS Lambda 上で完結する必要があり、実行環境は ephemeral で、従来型のストリーミング基盤は使えませんでした。

### 実施内容

* API Gateway と Lambda を使ったサーバーレス WebSocket ストリーミングを構築。
* Lambda 内で binary WebM payload を抽出し、音声を再構成。
* Lambda の warm-container reuse を活用し、短命な会話コンテキストを維持。
* コンテナ再利用時のデータ損失を防ぐため、セッション終了時の永続化保護を追加。

## 課題 #2

### 単一の AI プロバイダでは要件を満たせなかった

評価の結果は次の通りでした。

```text
Azure
↓
日本語認識に強い

AWS
↓
話者分離に強い
```

どちらか一方だけでは、業務要件を満たせませんでした。

### 実施内容

ハイブリッドクラウドの "dual-brain" アーキテクチャを設計しました。

Azure が担当:

* 音声認識
* 意図解析

AWS が担当:

* 話者識別
* 会話構造化

これにより、各プロバイダの強みを活かしつつ、下流側は一貫した処理モデルで扱えるようにしました。

## 課題 #3

### クロスクラウドでのタイムライン整合

統合後に最も難しかったのは、時間軸の整合でした。

Azure と AWS は異なる時間モデルを使っていました。

```text
Azure
100ns Tick

AWS
Seconds
```

さらに、無音処理の違いによって、およそ:

```text
3.5 秒
```

のタイムラインドリフトが発生しました。

補正しなければ:

* 話者属性が崩れる
* 文字起こしの所有権が不安定になる
* レコメンド品質が落ちる

### 実施内容

タイムライン整合エンジンを実装し、次を実現しました。

* 異なるタイムスタンプ形式の正規化
* プロバイダ固有のドリフト補正
* 文字起こし区間と話者区間の統計的マッチング
* 会話の所有権を再構成

これにより、会話全体を通じて顧客と営業担当の発話を正確に属性付けできました。

## 信頼性工学

このプラットフォームには相当量の運用強化が必要でした。

### DB 復旧

次を改善しました。

* コネクションリーク
* ロック待ちタイムアウト
* 長時間トランザクションの停止

接続ライフサイクルとトランザクション管理の改善で解消しました。

### イベント駆動の分離

非同期処理を導入しました。

```text
音声アップロード
↓
S3 イベント
↓
文字起こし
↓
話者属性
↓
意図解析
↓
永続化
```

AI 処理ステージ間の結合度を下げ、耐障害性を改善しました。

### 失敗予防

次を実装しました。

* 冪等処理
* クォータ考慮の実行
* S3 トリガー・ループ防止

これにより、運用コストとリスクを抑えました。
