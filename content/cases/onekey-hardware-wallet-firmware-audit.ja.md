## エグゼクティブサマリ

この監査は、一般向けのハードウェアウォレット・ファームウェアを、マーケティングではなく工程の観点から評価したものです。

単一のバグというより、ビルドチェーン、Secure Element の境界、乱数の健全性、パス深度検証の各層に弱い点が重なっていました。

ハードウェアウォレットは「最後の防壁」です。だからこそ、各層で決定論的に信頼できなければ、本当の意味でのセキュリティ境界にはなりません。

## この事例の意味

ハードウェアウォレットの価値は、見た目の安全性ではなく、次の要素で決まります。

- 再現可能なビルド
- 迂回できない状態遷移
- 乱数の健全性確認
- プロトコルとメモリモデルの厳密な境界

この事例は、製品レベルでは安全そうに見えても、内部のアーキテクチャに危険が残ることを示しています。

## 1. ビルドチェーンの問題

最初の論点は決定性でした。

配布されたファームウェアが、ソースコードと曖昧さなく対応づけられる必要があります。隠れたリポジトリ構成、段階的で見えにくい配布、供給網の不透明さは、いずれも検証性を下げます。

求められるのは次の条件です。

- 単一の真実の参照元
- 再現可能なバイナリ
- 追跡可能なリリース状態
- ソースからデバイスまでの経路が明確であること

## 2. 状態機械の分断

次の論点は Secure Element の境界でした。

健全な設計では、Secure Element がポリシーを強制し、主 MCU がそれを黙って上書きできません。ところが監査対象では、MCU 側の論理状態が Secure Element の分離性を弱めうる構造がありました。

このような設計では、ハードウェア安全機能が単なる同期部品に落ちてしまいます。

## 3. 暗号工学上の負債

次の論点は乱数処理でした。

返却されたデータの長さだけを確認しても不十分です。安全性が重要なファームウェアでは、乱数が実際に健全かどうかの検証が必要です。

監査では、現代のハードウェアウォレットに残すべきではない古い暗号パターンや、暗号化されて見えても構造情報が漏れうる境界問題も確認されました。

## 4. 境界の不一致

最後の論点は、プロトコル深度とメモリ構造の不一致でした。

パーサーが受け入れる経路深度が、下位の C 構造体が安全に保持できる深さを超えていると、アウトオブバウンズの問題につながります。

これは言語の問題ではなく、契約の問題です。

## 主な指摘

- ビルドチェーンの決定性が安全性説明に追いついていない
- Secure Element の状態が MCU 論理に従属しうる
- 乱数の健全性検証が不足
- パス深度とバッファサイズの前提が不整合
- 境界チェックが上位層任せになっていた

## 実務上の示唆

ハードウェアウォレットの安全性は、単一機能では成立しません。

再現可能なビルド、決定論的なポリシー強制、検証済みの乱数、そして各層でのメモリ安全な契約が揃って初めて、セキュリティ境界として成立します。
