# 産業用テレメトリ 107億件のスケーリング

## エグゼクティブサマリ

再生可能エネルギー向けクラウド基盤の主担当として、プロトタイプ段階のシステムを、**107億件超**のデータを扱える産業グレードのテレメトリ基盤へ引き上げました。

改善対象は Elasticsearch だけではありませんでした。実際の成果は、アプリケーション挙動、ストレージ配置、検索のガバナンス、可視化性能、デプロイ再現性をまとめて見直したことから生まれました。

## 背景

このプラットフォームは、地理的に分散した風力発電所からテレメトリと監視データを集める、大規模な再生可能エネルギー運用基盤でした。

本番規模では、環境は次のレベルまで拡大していました。

```text
107億件超のテレメトリデータ
約700GBの Elasticsearch データセット
```

データ量が増えるにつれて、クエリ遅延は不安定になり、ストレージコストは上昇し、運用安定性の維持も難しくなっていました。もはや課題は機能追加ではなく、産業規模で性能を維持することでした。

## 課題

### 大規模化による検索性能劣化

Elasticsearch クラスタが拡大するにつれて、繰り返し調整しても応答時間は悪化していきました。

一般的な調整対象は次の通りでした。

* Elasticsearch パラメータ
* JVM 設定
* クエリ最適化

ただし、これらの改善は限定的でした。

根本ボトルネックは解消されていませんでした。

## 調査

Elasticsearch を単独コンポーネントとしてではなく、リクエスト経路全体として分析しました。

```text
アプリケーション
↓
Elasticsearch
↓
仮想化レイヤ
↓
ストレージサブシステム
↓
物理 I/O
```

Esrally と fio を使って本番相当のワークロードを再現しました。

結論は明確でした。主たるボトルネックは Elasticsearch ではなく、ストレージ遅延でした。

## 根本原因

ランダム I/O の負荷下では、仮想化ストレージが大きなペナルティを生んでいました。

いくつかのケースでは、仮想化レイヤの遅延は、同等の直結 SSD 構成に対して **最大 20 倍** 悪化していました。

このため、何度 Elasticsearch を調整しても意味のある改善が出ませんでした。

検索エンジンがストレージ待ちになっていたのです。

## 実施内容

### ストレージを意識した構成

理論上のスループットではなく、実際のワークロード特性でインフラを分類しました。

用途ごとに分けて設計しました。

* ホット検索
* テレメトリ取り込み
* 履歴アーカイブ

これにより、すべての層を同じものとして扱うのではなく、アクセスパターンに合わせた設計へ変えました。

### Elasticsearch のガバナンス

インデックスとシャード割り当てを再設計し、運用予測性を高めました。

主な目的は以下です。

* シャード分散の平準化
* JVM 負荷の低減
* ノード復旧の高速化
* クラスタ安定性の向上

### 可視化のスケーリング

データ量の増加に伴い、既存の監視 UI も別のボトルネックになっていました。

数万点しか扱えなかった描画を、数十万点まで実用的に拡張し、オペレータの応答性を維持しました。

### プラットフォームの近代化

コンテナ化と構成簡素化でデプロイ手順を標準化しました。

結果:

* 運用コストを約 85% 削減
* 環境差分を縮小
* デプロイ再現性を向上
* 提供速度を改善

## 成果

| 指標 | 結果 |
| --- | --- |
| テレメトリデータ量 | 107億件超 |
| Elasticsearch データ量 | 約700GB |
| クエリ遅延 | 数秒 -> 65〜137ms |
| ストレージ起因の遅延 | 仮想化で最大 20 倍 |
| 可視化性能 | 数万点 -> 数十万点 |
| 運用コスト | 約 85% 削減 |

## 学び

大規模システムは、単一コンポーネントの失敗だけでは壊れません。

最も大きな改善は、層間の関係を理解したことから生まれました。

```text
アプリケーションロジック
↓
検索基盤
↓
仮想化
↓
ストレージ
↓
物理ハードウェア
```

この案件が示したのは、持続的なスケーラビリティは個別チューニングではなく、層を跨いだ性能工学で実現するということです。

## この事例の意味

これは単なる Elasticsearch のチューニング案件ではありません。

実際には次を跨ぐ、クロススタックの性能調査でした。

* 大規模データ基盤
* ストレージを意識したアーキテクチャ
* インフラのスケーラビリティ
* 性能工学
* 運用信頼性

同じ方法は、現代のデータ基盤、分散システム、クラウド基盤、AI データパイプラインにもそのまま適用できます。
