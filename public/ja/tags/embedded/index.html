8KB RAM環境におけるSM9(IBC)暗号の実装

挑戦：ベアメタル上におけるIBCの「恐怖」 2013年当時、ARM Cortex-M3 (STM32F2) 上で SM9 (Identity-Based Cryptography) 規格を実装することは、多くのエンジニアにとって「不可能」と考えられていました。標準的なECCとは異なり、SM9は**ペアリング（Pairing-friendly curves）**を必要とし、256ビットの座標を1536ビットの構造に拡張する大規模な体拡大（Field Expansion）を伴うためです。 標準的な暗号ライブラリ（OpenSSL, MIRACL）はサーバー向けに設計されており、RAMの1バイトが貴重な組込みシステムには適していません。私の任務は、肥大化した実装を徹底的に削ぎ落とし、セキュアUSBKeyやオフラインATMダイナミックロックシステムに求められる 8KB RAM以下 の制約内に収めることでした。 1. 外科的アーキテクチャ：「メタデータの肥大化」の除去 2013年の監査で最初に発見したのは、メモリフットプリントの70%がアルゴリズムそのものではなく、インフラストラクチャによるものだということでした。 ASN.1 形式のデコード: ヒープ空間を大量に消費する要因。 OpenSSL への依存: マイクロコントローラには不要な数千行のコード。 戦略：「純粋アルゴリズム」の抽出 私は実装をベアメタルCプリミティブへと絞り込む作業を主導しました。複雑なASN.1を廃止し、Protobuf または静的な固定サイズ構造へと切り替えました。これにより「メモリドリフト」のリスクを排除し、ヒープのフットプリントを60%以上削減しました。 2. メモリの制御：mralloc.c によるブレークスルー 8KBの制限を達成するために、標準の malloc に頼ることはできませんでした。私は基盤となる MIRACL メモリアロケータ（mralloc.c）を改造し、ピーク時のヒープ使用量を厳密に統計・追跡する機能を組み込みました。 /* * 2014年 8KB RAM最適化作業からのスニペット * 目的: ECDH/SM9 鍵交換におけるピークメモリの制限 */ static size_t peak_memory_usage = 0; static size_t current_usage = 0; void* mr_alloc(size_t size) { current_usage += size; if (current_usage > peak_memory_usage) { peak_memory_usage = current_usage; } // リソース制約のあるターゲットに対する厳格な執行 if (peak_memory_usage > MAX_RAM_BUDGET_BYTES) { log_critical("メモリ制限違反！ 予算を超過しました: %zu", peak_memory_usage); return NULL; } return standard_malloc(size); } 結果: 徹底的な洗練の結果、フルスタックのECDHでピーク時のアロケーションを 9,300バイト まで抑えることに成功しました。「科学者」としての視点を持ってメモリを外科的に管理することで、汎用ライブラリを遥かに凌駕する性能を引き出せることを証明しました。 ...

USB HIDプロトコル：127バイトの『幽霊』と消えた先頭バイトの解決

USB HIDプロトコル・エンジニアリング：64バイトの制約を超えて 電子暗号錠などの高セキュリティ・ハードウェアを設計する際、物理レイヤー（USB HID）の厳格な制約は、現代的なデータ形式（JSON/XML）としばしば衝突します。本プロジェクトでは、USB HIDのための独自の**断片化・再構成（F&R）**プロトコルの構築と、OSドライバ・スタックに潜む致命的なバグの解決を行いました。 1. 断片化・再構成（F&R）プロトコルの設計 USB HIDは物理的に64バイトのパケット（レポート）に制限されています。700バイトを超えるセキュリティ・マニフェストを転送するため、アプリケーション・レイヤーに以下のプロトコルを実装しました： 6バイトの制御ヘッダー: Total Length (16-bit): 再構成後の論理メッセージの総サイズ。 Current Offset (16-bit): グローバル・バッファ内での現在の断片の位置。 Block Index (16-bit): 到着順序を検証するためのシーケンス番号。 58バイトのペイロード: 各フレームは最大58バイトの実データを運び、100%決定論的なパケット組み立てを保証します。 当初は8ブロックに制限されていましたが、バッファを12ブロックに拡張し、マルチスレッド環境での競合を防ぐためにセッション・ハンドルを導入。高負荷なストレス・テスト下でもデータの整合性を維持しました。 2. 「生贄のバイト」ハック：Windowsドライバとの戦い 大手国有インフラ企業との統合テスト中に、「メッセージの先頭バイトが消失する」という不可解な現象に直面しました。ホストが6バイトのヘッダーを送信しても、MCU側では常に5バイトしか受信できないという状態でした。 根本原因 WindowsのHIDドライバ・スタックを深く調査した結果、隠された仕様が判明しました。OSは制御転送の第0バイトをReport IDとして解釈します。独自のファームウェアでReport IDを定義していなかったため、ドライバがヘッダーの最初の1バイトを勝手に「飲み込んで」いたのです。 解決策：65バイト・バッファ ファームウェアの修正が困難な状況であったため、ホスト側で「外科的」な解決策を講じました： ホストの送信バッファを64から65バイトに拡張。 第0バイトにダミーとなる 0x00 バイトを配置。 Windowsドライバがこの 0x00 をReport IDとして「消費」することで、実際の64バイトのデータ構造（第1バイトから開始）は無傷でハードウェアに届き、完璧なデータアライメントが実現しました。 3. 「127バイトの幽霊」：ASN.1 エンコーディングの越境 また、ある大手銀行のレガシー端末から取得した14MBの16進数ダンプを詳細に解析した際、断続的な認証失敗の原因となっていた「幽霊バグ」を特定しました。 プロトコル・スタックのバイナリ・リバースエンジニアリングを通じて、ASN.1 長さフィールドのオーバーフローが127バイトの境界で発生していることを突き止めました。BER/DERエンコーディングでは、127バイトを超える長さは複数バイト形式を必要としますが、レガシーなミドルウェアがこの長さビットを誤って切り捨てていたため、127バイトを超える全ペイロードの復号に失敗していたのです。 ミドルウェア内にカスタム・プロトコル・バリデーターを実装し、これらの不正なASN.1構造を正規化してからコアロジックに渡すことで、銀行のホスト側ソフトウェアを書き換えることなくシステムを安定化させました。 主な成果 100%のデータ整合性: OSとハードウェアの境界線上で、700バイト超のセキュアコマンドのゼロパケットロスを達成。 レガシー・システムの救済: 10年間「原因不明」とされていた国内大手銀行の端末エラーを、127バイトの境界条件の特定により解決。 事業への貢献 (ROI): ハードウェアの回収やファームウェアの大規模更新を回避し、ホスト側のハックとバリデーターの実装のみで数億円規模の損失を防ぎました。 [!TIP] アーキテクトの視点: 低レイヤー・エンジニアリングにおいて、「真実」はコードではなくドライバの中に隠れていることがあります。ハードウェアとソフトウェアの意見が食い違うとき、アーキテクトはデジタル考古学者のように深層を掘り下げなければなりません。
