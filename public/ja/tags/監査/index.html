ディープダイブ：RWAトークン化プロトコルにおけるアーキテクチャの欠陥

はじめに 2026年第1四半期、私は東京を拠点とするコンプライアンス重視のWeb3スタートアップから、緊急のアーキテクチャ監査の依頼を受けました。このプロジェクトは、現物資産をERC-1155トークンとして分割し、不動産市場に革命を起こすことを目的としていました。 表面上、プラットフォームは洗練されたUIを備え、一見「MVP（最小実行可能製品）」として機能しているように見えました。しかし、その内部構造は「紙の防弾チョッキ」のようなものでした。見た目は堅牢ですが、保護すべき数百万ドルのTVL（預かり資産）を守る能力は根本的に欠如していました。 私が発見したのは単なるいくつかのバグではなく、論理ミス一つが即座に財務的破綻に直結する分野における、**エンジニアリング・ディシプリン（規律）**の構造的な欠如でした。 1. 「1ユニット」価格の脆弱性（プロトコルレベルの破綻） 最も深刻なP0脆弱性は、AssetFactory.solコントラクトの主要な購入関数に存在しました。このロジックは、ユーザーが不動産の分割所有権を購入できるように設計されていました。 脆弱性の内容 コントラクトは、1ユニットあたりの「価格（cost）」を正しく取得していました。しかし、根本的な乗算処理、つまり「総支払額が要求された数量に見合っているか」のチェックを怠っていました。 攻撃シナリオの図解 [ 攻撃者 ] ⮕ purchase(propertyId=1, amount=10,000) を呼び出し │ ▼ [ AssetFactory.sol ] │ ├─ 取得: ユニット単価 = 0.1 ETH │ ├─ チェック: msg.value == ユニット単価? (✅ 0.1 ETH が提供された) │ ⚠️ 警告: 単価 × 数量 (1,000 ETHが必要) のチェックを失念！ │ ├─ 状態: トランザクション承認 ✅ │ └─ アクション: 10,000ユニットを攻撃者にミント ⮕ [在庫が枯渇] 単価1ビット分のコストを支払うだけで、攻撃者は高価値不動産資産の全在庫を奪い取ることが可能でした。これは、文脈（意図）を理解できない自動AIスキャナーが見落としがちな、人間のアーキテクトが数秒で見抜くべき「ビジネスロジックの破綻」の典型例です。 ...
